# 프로듀서 신뢰성

## 핵심 원칙

브로커에 높은 신뢰성을 설정해도, 프로듀서가 올바르게 설정되지 않으면 메시지가 유실될 수 있다. 개발 시 반드시 신경써야 할 두 가지:

1. 신뢰성 요구 조건에 맞는 올바른 `acks` 설정
2. 설정과 코드 모두에서 에러를 올바르게 처리

## acks 설정 상세

| acks | 응답 시점 | 데이터 유실 위험 | 지연 |
|------|----------|----------------|------|
| `0` | 네트워크 전송 직후 | 매우 높음 (브로커 수신 확인 없음) | 최저 |
| `1` | 리더 쓰기 완료 후 | 중간 (팔로워 복제 전 리더 크래시 시) | 중간 |
| `all` (-1) | 모든 ISR 쓰기 완료 후 | 가장 낮음 | 가장 높음 |

### acks=0
- 파티션 오프라인, 리더 선출 중, 전체 클러스터 불능 상태에서도 에러를 받지 않음
- 지연은 낮지만, 종단 지연은 개선되지 않음 — 컨슈머는 HW가 갱신되어야 읽을 수 있기 때문

### acks=1
- 리더가 파티션 데이터 파일에 쓴 직후 응답 (디스크 반영은 불필요)
- **유실 시나리오**: 리더에 쓰여졌으나 팔로워 복제 완료 전 리더 크래시 → 새 리더는 해당 메시지를 모름
- 불완전 복제 파티션(under-replicated partition) 발생 가능 — 리더가 복제 전에 먼저 응답하기 때문

### acks=all
- 리더가 모든 ISR이 메시지를 받을 때까지 대기 후 응답
- `min.insync.replicas`와 함께 사용해 최소 복제 수를 보장
- 가장 안전하나 지연이 가장 큼

## 프로듀서 재시도 설정

### 에러 유형

| 에러 유형 | 예시 | 처리 방법 |
|----------|------|----------|
| **재시도 가능** | `LEADER_NOT_AVAILABLE` | 자동 재시도 |
| **재시도 불가능** | `INVALID_CONFIG`, 메시지 크기 초과, 인가 에러 | 즉시 실패 처리 |

### 권장 재시도 설정

```properties
# 재시도 횟수를 사실상 무한으로 설정
retries=MAX_INT

# 포기 전 최대 대기 시간으로 기준 삼기
delivery.timeout.ms=120000  # 예: 2분
```

`delivery.timeout.ms` 내에서 계속 재시도하도록 설정하는 것이 권장 방식이다.

### 재시도와 중복의 위험

재시도는 **최소 한 번(at-least-once)** 전달을 보장하지만 정확히 한 번(exactly-once)은 보장하지 않는다.

- 브로커에 정상적으로 쓰였지만 프로듀서가 ack를 받지 못해 재전송 → 중복 발생 가능

### 멱등성 프로듀서

```properties
enable.idempotence=true
```

- 프로듀서가 레코드에 추가 시퀀스 정보를 포함
- 브로커가 재시도로 인한 중복 메시지를 자동으로 건너뜀
- 재시도로 인한 중복 방지 → 정확히 한 번 전달에 가까워짐

## 추가 에러 처리

자동 재시도가 처리하지 못하는 에러를 개발자가 직접 처리해야 한다:

| 에러 상황 | 처리 방법 |
|----------|----------|
| 재시도 불가능 브로커 에러 (메시지 크기, 인가) | 즉시 폐기 또는 로깅 |
| 직렬화 에러 (브로커 전달 전 발생) | 개발자 핸들러 필요 |
| 재시도 소진 또는 메모리 부족 | 로컬 디스크 저장, 백프레셔, 원본 시스템 중단 |
| 타임아웃 | 재전송 또는 폐기 결정 |

에러 핸들러가 단순 재전송만 수행한다면 프로듀서 재시도 기능을 그대로 쓰는 것이 더 낫다.

## acks=1 유실 시나리오 (상세)

```
acks=1, replication.factor=3, unclean.leader.election=false 환경

1. 프로듀서 → 리더에 메시지 쓰기 완료
2. 리더 → 프로듀서에 성공 응답 전송
3. 리더 크래시 (팔로워에 아직 복제 안 됨)
4. 팔로워가 아직 out-of-sync 판정을 받지 않아 ISR로 간주됨
5. 팔로워 중 하나가 새 리더로 선출
6. 해당 메시지는 새 리더에 없음 → 유실
7. 컨슈머 입장에서는 본 적 없는 메시지 → 일관성은 유지
```
