# 컨트롤러

## 개념

컨트롤러는 일반 카프카 브로커 기능에 더해 **파티션 리더 선출** 역할을 추가로 담당하는 브로커다. 클러스터에는 항상 정확히 하나의 컨트롤러만 존재한다.

## 컨트롤러 선출 (ZooKeeper 방식)

1. 가장 먼저 시작된 브로커가 ZooKeeper `/controller` 경로에 Ephemeral 노드 생성 → 컨트롤러가 됨
2. 나머지 브로커들은 "노드가 이미 존재함" 예외를 받고, 해당 노드에 **watch** 설정
3. 컨트롤러 노드 삭제 시 (종료, 세션 타임아웃, 연결 끊김) 브로커들이 알림을 받아 경쟁적으로 새 노드 생성
4. 가장 먼저 성공한 브로커가 다음 컨트롤러

## 에포크와 좀비 컨트롤러

- 새 컨트롤러 선출마다 ZooKeeper 조건적 증가 연산으로 **에포크(epoch)** 증가
- 모든 브로커는 현재 컨트롤러 에포크를 알고 있으며, **낮은 에포크의 메시지를 무시**
- **좀비 컨트롤러**: GC로 멈춘 구 컨트롤러가 새 컨트롤러 선출을 모르고 브로커에 메시지를 보내는 현상. 에포크 검증으로 방어.

## 파티션 리더 선출 프로세스

브로커 이탈 감지 시:

1. 해당 브로커가 리더를 맡던 파티션 목록 파악
2. 각 파티션 레플리카 목록에서 **다음 레플리카**를 새 리더로 결정
3. 새 상태를 ZooKeeper에 기록 (비동기 배치 방식)
4. 영향받는 모든 브로커에 **LeaderAndISR 요청** 전송 (새 리더/팔로워 정보 포함)
5. 클러스터 전체 브로커에 **UpdateMetadata 요청** 전송 (MetadataCache 갱신)

## MetadataCache

모든 브로커가 유지하는 클러스터 전체 브로커·레플리카 맵. 클라이언트의 메타데이터 요청 응답에 사용된다.

## KRaft: 래프트 기반 새 컨트롤러

### ZooKeeper 방식의 문제점

1. 메타데이터 쓰기(동기) ↔ 브로커 메시지 전송(비동기) 불일치 → 메타데이터 불일치 가능
2. 컨트롤러 재시작 시 ZooKeeper에서 전체 메타데이터 읽기 + 모든 브로커에 전송 → 파티션/브로커 수 증가 시 병목
3. 메타데이터 소유권 분산 (컨트롤러/브로커/ZooKeeper가 각기 다른 작업 담당)
4. 운영 복잡성: 카프카와 ZooKeeper 두 분산 시스템을 동시에 관리해야 함

### KRaft 핵심 아이디어

컨트롤러 노드들이 **메타데이터 이벤트 로그를 관리하는 래프트 쿼럼**이 됨:

- 토픽, 파티션, ISR, 설정 등 모든 메타데이터를 **이벤트 로그**로 저장
- 래프트 알고리즘으로 외부 시스템 없이 자체적으로 리더(**액티브 컨트롤러**) 선출
- 팔로워 컨트롤러들은 액티브 컨트롤러 데이터를 복제 → 장애 시 즉시 투입 가능 (긴 리로드 불필요)

### ZooKeeper vs KRaft 비교

| 항목 | ZooKeeper 방식 | KRaft 방식 |
|------|---------------|------------|
| 메타데이터 저장 | ZooKeeper | 메타데이터 이벤트 로그 (컨트롤러 쿼럼) |
| 변경 전파 | 컨트롤러 → 브로커 (push) | 브로커 → 액티브 컨트롤러 (pull, MetadataFetch API) |
| 브로커 등록 | 시작/종료 시 ephemeral 노드 | 컨트롤러 쿼럼에 등록, 운영자가 명시적으로 해제 전까지 유지 |
| 컨트롤러 장애 복구 | 전체 메타데이터 리로드 필요 | 팔로워가 최신 상태 유지 → 즉시 투입 가능 |

### Fenced State

브로커가 온라인이지만 메타데이터가 최신이 아닌 경우 **fenced 상태**가 됨. 클라이언트 요청 처리 불가. 자신이 리더가 아님에도 리더라 착각하는 브로커의 잘못된 쓰기를 방지한다.

> KRaft는 Kafka 2.8 프리뷰, **3.3부터 프로덕션 사용 가능**
