# 멱등적 프로듀서

## 개념

동일한 작업을 여러 번 실행해도 한 번 실행한 것과 결과가 같은 성질을 **멱등성(Idempotence)**이라 한다.

```sql
UPDATE t SET x = x + 1 WHERE y = 5  -- 멱등적 X (호출할 때마다 결과가 다름)
UPDATE t SET x = 18 WHERE y = 5     -- 멱등적 O (몇 번 실행해도 결과 동일)
```

카프카 프로듀서를 `acks=all`로 설정해도 재전송 중 브로커 응답이 유실되면 중복 메시지가 발생한다. 멱등적 프로듀서는 이 경우를 자동으로 처리한다.

## 중복 발생 시나리오

```
1. 리더가 메시지를 받고 팔로워에 복제 성공
2. 리더가 프로듀서에 응답을 보내기 전 크래시
3. 프로듀서는 타임아웃 → 메시지 재전송
4. 재전송된 메시지가 새 리더에 도착 → 이미 저장된 메시지와 중복
```

## 활성화

```properties
enable.idempotence=true
```

- `acks=all`이 자동 적용
- `retries=MAX_INT`이 자동 적용
- `max.in.flight.requests.per.connection=5` 이하여야 함 (기본값이자 최대 허용값)

## 작동 원리

### 식별자 구조

| 구성요소 | 타입 | 설명 |
|---------|------|------|
| 프로듀서ID (PID) | long | 브로커가 초기화 시 발급 |
| 시퀀스 넘버 | int | 배치 단위로 단조 증가 |

**고유 식별자**: `대상 토픽 + 파티션 + 프로듀서ID + 시퀀스 넘버`

각 브로커는 파티션별로 **마지막 5개 시퀀스 넘버**를 메모리에 추적한다. 이미 받은 시퀀스 번호가 다시 오면 거부(로깅 + 지표 반영, 예외는 발생하지 않음).

### 오버헤드

- 프로듀서 시동 시 API 1회 추가 호출 (PID 발급)
- 각 레코드 배치에 96비트 추가 (PID + 첫 배치의 시퀀스 넘버)

### `out of order sequence number` 에러

브로커가 예상보다 높은 시퀀스 넘버를 받으면 이 에러가 발생한다. 트랜잭션 없이 멱등적 프로듀서만 사용하는 경우 이 에러는 보통 무시해도 되나, **프로듀서와 브로커 사이 메시지 유실**이 있었음을 의미하므로 로그에서 발견 시 설정을 점검해야 한다.

## 장애 시나리오

### 프로듀서 재시작

트랜잭션 없이 멱등적 프로듀서만 사용하면, 재시작 시마다 **새로운 PID**가 발급된다. 따라서 프로듀서 재시작 전후의 중복은 감지할 수 없다. 재시작 전 구 프로듀서가 보낸 메시지를 신 프로듀서가 재전송하면 브로커는 서로 다른 프로듀서의 메시지로 판단한다.

### 브로커 장애

팔로워가 리더가 될 때는 이미 최근 5개의 시퀀스 넘버를 메모리에 가지고 있다 (리더 복제 중 함께 업데이트됨). 따라서 장애 후에도 유효성 검증이 즉시 재개된다.

브로커 재시작 시에는:
1. 스냅샷 파일에서 프로듀서 상태 로드
2. 현재 리더로부터 복제한 레코드로 상태 업데이트
3. 크래시로 스냅샷이 최신이 아닌 경우, 로그 세그먼트의 메시지로 추가 복구

## 한계

| 상황 | 멱등적 프로듀서 방어 가능 여부 |
|------|-------------------------------|
| 프로듀서 내부 재시도로 인한 중복 | ✅ 방지 |
| 애플리케이션이 `send()`를 두 번 호출 | ❌ 방지 불가 |
| 여러 프로듀서 인스턴스가 동일 메시지 전송 | ❌ 방지 불가 |
| 프로듀서 재시작 후 이전 메시지 재전송 | ❌ 방지 불가 (트랜잭션 필요) |
