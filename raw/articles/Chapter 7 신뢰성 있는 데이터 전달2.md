# 7.4 신뢰성 있는 시스템에서 프로듀서 사용하기

브로커에 가장 높은 신뢰성 설정을 적용하더라도, 프로듀서 역시 신뢰성이 있도록 설정하지 않으면 시스템 전체적으로 데이터가 유실될 수 있다. 이것을 보여주는 두 가지 예를 살펴보자.

1. 토픽별로 3개의 레플리카를 가지도록 브로커를 설정한 상태에서 언클린 리더 선출 기능을 끈다.
    
    - 이렇게 되면 카프카 클러스터에 커밋된 메시지는 유실되지 않음
    - 하지만 프로듀서가 메시지를 보낼 때 `acks=1` 설정으로 보내도록 설정한다.
    - 프로듀서에서 메시지를 전송해서 리더에는 쓰여졌지만, 아직 ISR에 반영되지 않은 상태
    - 리더가 프로듀서에게 메시지가 성공적으로 쓰여졌다고 응답을 보낸 직후 크래시가 나서 데이터가 레플리카로 복제되지 않는다.
    - 다른 레플리카들은 여전히 인-싱크 상태로 간주되기 때문에(레플리카는 아웃-오브-싱크 상태로 판정될 때까지 약간의 시간이 걸림), 그중 하나가 리더가 됨
    - 메시지가 레플리카에 쓰여지지 않은 만큼 해당 메시지는 유실
    - 하지만 메시지를 쓰고 있는 애플리케이션 입장에서는 성공적으로 쓰여졌다 착각할 수 있음
    - 어떤 컨슈머도 이 메시지를 보지 못한 만큼(레플리카 입장에서는 본 적도 없는 메시지이기 때문에 커밋된 것도 아님), 시스템 일관성은 유지
    - 하지만 프로듀서 입장에서 메시지는 유실된 것
    
    <aside>
    
    즉, `acks=1` 인 경우 리더에 쓰여지면 성공으로 간주. 이후 브로커가 ack를 보낸 후 바로 크래시가 나면 다른 레플리카가 아직 복제되지 않았지만, 아웃-오브-싱크 상태로 판정되기 까지 약간의 시간이 걸리기 때문에 ISR로 판정되어 리더가 될 수 있다. 프로듀서 입장에서 새로 선출된 리더는 이전 리더의 메시지를 온전히 복제하지 않았지만, 컨슈머 입장에서는 본 적 없는 메시지이기 때문에 시스템 일관성은 유지된다.
    
    </aside>
    
2. 토픽별로 3개의 레플리카를 가지도록 브로커를 설정한 상태에서 언클린 리더 선출 기능을 끈다.
    
    - 이번에는 `acks=all` 설정으로 메시지 전송
    - 카프카에 메시지를 쓰려고 있는데, 쓰고 있는 파티션의 리더 브로커가 방금 전 크래시 나서 새 리더는 아직 선출중이라 가정
    - 카프카는 “Leader not Available” 응답을 보냄
    - 이 시점에서 만약 프로듀서가 올바르게 에러를 처리하지 않고 쓰기가 성공할 때까지 재시도도 하지 않을 경우 메시지는 유실될 수 있음
    - 이것은 브로커의 신뢰성 문제와는 별개의 문제. 즉, 브로커는 메시지를 받지도 않았기 때문
    - 컨슈머 역시 메시지를 받은 적이 없기 때문에 일관성 문제도 아님
    - 하지만 프로듀서가 이 에러를 정확히 처리해주지 않을 경우, 메시지 유시을 초래
    
    <aside>
    
    즉, 리더를 선출 중인 상황에서 프로듀서가 이 에러를 처리하지 않고, 쓰기가 성공할 때까지 재시도 하지 않으면 메시지가 유실될 수 있다. 이 때 브로커는 메시지를 받은 적이 없기 때문에 브로커의 신뢰성과 메시지 유실과는 관계가 없게 된다.
    
    </aside>
    

이 예제들에서 볼 수 있듯, 카프카에 메시지를 쓰는 애플리케이션을 개발할 때 신경써야 할 중요한 것이 두 가지 있다.

1. 신뢰성 요구 조건에 맞는 올바른 acks 설정을 사용
2. 설정과 코드 모두에서 에러를 올바르게 처리

## 7.4.1 응답 보내기

1. `acks=0`
    - 프로듀서가 네트워크로 메시지를 전송한 시점에 메시지가 카프카에 성공적으로 쓰여진 것으로 간주
    - 전송하는 객체가 직렬화될 수 없거나 네트워크 카드가 오작동할 경우 에러를 받지만, 파티션이 오프라인, 리더 선출중, 전체 카프카 클러스터 작동 불능인 경우 에러가 발생하지 않음
    - 이 설정은 지연은 낮지만, 종단 지연이 개선되지 않음
        - 컨슈머는 하이워터마크 갱신(모든 인-싱크 레플리카로 복제되지 않으면 HW가 생신되지 않음)되어야 메시지를 읽을 수 있음
2. `acks=1`
    - 리더가 메시지를 받아서 파티션 데이터 파일에 쓴 직후 응답 또는 에러를 보낸다는 것을 의미
        - 디스크 반영될 필요는 없음(페이지 캐시에 저장된 경우엔 응답)
    - 일부 메시지가 리더에 성공적으로 쓰여져서 클라이언트로 응답이 간 상태에서 미처 팔로워로 복제가 완료되기 전에 리더가 정지하거나 크래시 날 경우 데이터가 유실될 수 있음
    - 이 설정을 사용할 경우 메시지 복제 속도보다 더 빨리 리더에 쓸 수 있기 때문에 불완전 복제 파티션(under-replicated partition)이 발생할 수 있음
    - 리더 입장에서 복제 완료 전에 프로듀서에게 먼저 응답하기 때문
3. `acks=all`
    - 리더가 모든 ISR이 메시지를 받아갈 때까지 기다렸다가 응답하거나 에러를 보낸다는 것을 의미
    - 브로커의 `min.insync.replicas` 설정과 함께, 이 설정은 응답이 오기전까지 얼마나 많은 레플리카에 메시지가 복제될 것인지 조절할 수 있게 해줌
    - 가장 안전한 옵션이기도 한데, 프로듀서는 메시지가 완전히 커밋될 때까지 계속해서 메시지를 재전송
    - 이 옵션은 프로듀서의 지연이 가장 길어지는 옵션으로 프로듀서는 모든 ISR이 메시지를 받을 때까지 기다린 뒤에 해당 메시지 배치에 완료 표시를 하고 작업을 진행

## 7.4.2 프로듀서 재시도 설정하기

- 프로듀서의 에러 처리는 두 부분으로 나뉨
    - 프로듀서가 자동으로 처리해주는 에러
    - 프로듀서 라이브러리를 사용하는 개발자들이 처리해야 하는 에러
- 프로듀서는 재시도 가능한 에러를 처리할 수 있음
- 프로듀서가 브로커에 메시지를 전송하면 브로커는 성공 혹은 에러 코드를 리턴
    - 에러 코드는 전송을 재시도하면 해결될 수 있는 것과 아닌 것 두 가지로 나뉨
- 브로커가 `LEADER_NOT_AVAILABLE` 에러 코드를 리턴할 경우, 프로듀서는 전송을 재시도할 수 있음
    - 아마도 새로운 브로커가 리더로 선출된 상황에서 두 번째 시도는 성공
    - `LEADER_NOT_AVAILABLE` 는 재시도 가능한 에러
- 만약 브로커가 `INVALID_CONFIG` 예외를 리턴할 경우, 같은 메시지를 재전송한다고 해서 설정이 변경되지 않음
    - 재시도 불가능 에러
- 메시지가 유실되지 않는 것이 목표일 경우, 가장 좋은 방법은 재시도 가능한 에러가 발생했을 경우 프로듀서가 계속해서 메시지 전송을 재시도하도록 설정하는 것
- 재시도에 관한 가장 좋은 방법은 3장에서 권장한 것과 같이 재시도 수를 기본 설정값(MAX_INT 사실상 무한)으로 설정하고 메시지 전송을 포기할 때까지 대기할 수 있는 시간을 지정하는 `delivery.timeout.ms` 설정값을 최대로 잡아 주는 것
- 프로듀서는 이 시간 간격 내에 있는 한, 메시지 전송을 계속 재시도함
- 전송 실패한 메시지를 재시도하는 것은 두 메시지(실패했다고 생각했지만 실제로는 아닌 메시지와 재전송된 메시지)가 모두 브로커에 쓰여지는, 결과적으로 메시지가 중복될 위험을 내포
    - 브로커에 정상적으로 쓰여졌지만, 프로듀서가 ack 응답을 받지 못해 재시도하는 경우에 발생
- 재시도와 주의 깊은 에러 처리는 각 메시지가 ‘최소 한 번’ 저장되도록 보장할 수 있지만, ‘정확히 한 번’은 보장 못함
- `enable.idempotence=true` 설정으로 프로듀서가 추가적인 정보를 레코드에 포함할 수 있도록, 이를 활용해 브로커가 재시도로 인해 중복된 메시지를 건너뛸 수 있도록 할 수 있음

## 7.4.3 추가적인 에러 처리

- 프로듀서의 재시도 기능을 사용하는 것은 메시지 유실 없이 다양한 종류의 에러를 올바르게 처리할 수 있는 쉬운 방법
- 개발자 입장에서는 다른 종류의 에러들 역시 처리할 수 있어야 하고, 다음과 같은 것들이 포함
    - 메시지 크기에 관련되었거나 인가 관련 에러와 같이 재시도 불가능한 브로커 에러
    - 메시지가 브로커에 전송되기 전에 발생한 에러(직렬화 과정에서 발생한 에러)
    - 프로듀서가 모든 재전송 시도를 소진했거나, 재시도 과정에서 프로듀서가 사용하는 가용 메모리가 가득 차 발생하는 에러
    - 타임아웃
- 3장에서 동기/비동기적 전송 방식 모두에 대해 에러 핸들러를 작성하는 방법에 대해서 살펴보았음
- 에러 핸들러들의 내용은 애플리케이션과 그 목표에 달려있는데
    - ‘잘못된 메시지’는 그냥 폐기해야 하는가?
    - 에러를 로깅해야 하는가?
    - 원본 시스템으로부터 메시지를 읽는 것을 중단해야 하는가?
    - 원본 시스템에 백프레셔를 적용해 잠시 메시지 전송을 멈추도록 해야 하는가?
    - 메시지를 로컬 디스크의 디렉토리에 저장해야 하는가?
- 이런 결정들은 아키텍처와 제품의 요구 조건에 따라 달라짐
- 만약 메시지 재전송이 에러 핸들러가 하는 일의 전부라면, 프로듀서의 재전송 기능을 사용하는 편이 더 낫다는 점을 유념

# 7.5 신뢰성 있는 시스템에서 컨슈머 사용하기

컨슈머는 커밋된 데이터만 읽을 수 있다. 즉, 모든 인-싱크 레플리카에 쓰여진 다음부터 읽을 수 있다. 즉, 컨슈머는 일관성이 보장되는 데이터만 읽는다. 컨슈머가 해야 할 일은 어느 메시지까지 읽었고 어디까지는 읽지 않았는지를 추적하는 것이다. 이것은 메시지를 읽는 도중에 누락되지 않게 하기 위해 필수적이다.

컨슈머는 파티션으로부터 데이터를 읽어 올 때 배치 단위로 읽어온 뒤 배치별로 마지막 오프셋을 확인하고, 브로커로부터 받은 마지막 오프셋 값에서 시작하는 다른 메시지 배치를 요청한다. 이로인해 컨슈머는 메시지 누락 없이 새로운 데이터를 올바른 순서로 읽어올 수 있다.

특정 컨슈머가 정지하면, 다른 컨슈머 입장에서는 어디서부터 작업을 재개해야 할지 알아야 한다. 즉, 이전 컨슈머가 정지하기 전에 마지막으로 읽은 오프셋은 무엇인가? ‘다른’ 컨슈머는 새로운 컨슈머일 수 있고, 기존 컨슈머가 재시작한 것일 수 있다. 어떤 컨슈머건 간에 정지한 컨슈머가 읽던 파티션을 가져갈 것이고 어느 오프셋부터 작업을 시작해야 할지 알아야 할 것이기 때문이다. 컨슈머가 읽어온 오프셋을 ‘커밋’해야하는 이유이다. 읽고 있는 파티션에 대해 어디까지 읽었는지 저장해 둬야 해당 컨슈머나 다른 컨슈머가 재시작한 뒤에도 어디서부터 작업을 계속해야 할 지 알 수 있기 때문이다. 컨슈머가 메시지를 누락할 수 있는 경우는 읽긴 했지만 아직 처리는 완료되지 않은 이벤트의 오프셋을 커밋하는 경우다. 이렇게 되면 다른 컨슈머가 작업을 물려받았을 때 이 메시지들은 건너뛰게 되므로 영원히 처리되지 않는다. 오프셋이 언제 어떻게 커밋되는지에 대해 신경써야 하는 것이 중요한 이유다.

<aside>

**커밋된 메시지 vs 커밋된 오프셋**

커밋된 메시지: 모든 ISR에 쓰여져 컨슈머가 읽을 수 있는 메시지를 의미

커밋된 오프셋: 컨슈머가 특정 파티션 어느 오프셋까지의 모든 메시지를 받아서 처리를 완료했는지를 알리기 위해 카프카에 보낸 오프셋

</aside>

## 7.5.1 신뢰성 있는 처리를 위해 중요한 컨슈머 설정

우리가 원하는 수준의 신뢰성을 갖는 컨슈머를 설정하기 위해 알아야할 컨슈머 속성은 네 개가 있다.

### 1. `group.id`

- 같은 그룹ID를 갖는 두 개의 컨슈머가 같은 토픽을 구독할 경우, 각각의 컨슈머에는 해당 토픽 전체 파티션의 서로 다른 파티션이 할당되므로 각각은 서로 다른 파티션의 메시지만 읽게 됨
- 컨슈머가 구독한 토픽의 모든 메시지를 읽어야 한다면 고유한 그룹 ID가 필요

### 2. `auto.offset.reset`

- 커밋된 오프셋이 없을 때, 컨슈머가 브로커에 없는 오프셋을 요청할 때 컨슈머가 어떻게 해야 할지를 결정
- `earliset` 를 사용하면, 파티션의 맨 앞에서부터 읽기 시작
    - 컨슈머는 많은 메시지를 중복 처리하게 되지만, 데이터 유실은 최소화
- `latest` 를 사용하면, 컨슈머는 파티션의 끝에서부터 읽기 시작
    - 중복 처리는 최소화하지만, 컨슈머가 일부 메시지는 누락할 것이 거의 확실

### 3. `enable.auto.commit`

- 일정한 시간에 컨슈머가 자동으로 오프셋을 커밋하게 할 것인지, 코드에서 직접 오프셋을 커밋하게 할 것인지 설정하는 옵션
    
- 자동 오프셋 커밋
    
    - 이점은 폴링 루프에서 모든 읽어온 레코드에 대한 처리를 하는 와중에, 우리가 처리하지 않은 오프셋을 실수로 커밋하는 사태가 발생하지 않도록 보장
    - 단점은 메시지 중복 처리를 우리가 제어할 수 없다는 점. 읽어온 메시지 중 일부만 처리했고 아직 자동 커밋이 되지 않은 상태에서 컨슈머가 멈추면, 컨슈머를 다시 시작했을 때 메시지 중복 처리를 피할 수 없음
    
    <aside>
    
    문제 1: 중복 처리 (Duplicate Processing)
    
    poll() → 오프셋 100~200 수신 → 처리 중 → [크래시] → 커밋 안 됨 재시작 → 마지막 커밋 오프셋(100)부터 다시 읽기 → 100~200 재처리
    
    커밋 주기(5초) 내에 크래시가 발생하면 마지막 커밋 이후 메시지가 중복 처리된다. auto.commit.interval.ms를 줄여도 중복을 줄일 수 있을 뿐 완전히 없애는 것은 불가능하다.
    
    문제 2: 처리 완료 전 커밋 (Silent Data Loss)
    
    poll() → 오프셋 100~200 수신 → [5초 경과] → poll() 호출 → 이전 poll 오프셋(200) 자동 커밋 → 처리 도중 예외 발생 → 오프셋은 이미 200 → 100~200 일부 메시지 **누락**
    
    자동 커밋은 처리 완료 여부를 모른다. 커밋 주기가 도래하면 처리 중인 메시지도 커밋된 것으로 표시되기 때문에, 이후 예외로 처리가 실패해도 해당 메시지를 다시 읽을 수 없다.
    
    커밋 주기(5초) 내에 크래시가 발생하면 마지막 커밋 이후 메시지가 중복 처리된다. auto.commit.interval.ms를 줄여도 중복을 줄일 수 있을 뿐 완전히 없애는 것은 불가능하다.
    
    </aside>
    
- 애플리케이션이 백그라운드에서 처리를 수행하기 위해 다른 스레드에 레코드를 넘기는 것과 같이 복잡한 처리를 해야 할 경우, 직접 오프셋을 커밋해주는 것 외에는 선택지가 없음
    
- 자동 커밋이 컨슈머가 읽어오긴 했지만 아직 처리하지 않은 오프셋을 커밋할 수도 있기 때문
    

### 4. `auto.commit.interval.ms`

- 자동 오프셋 커밋을 사용할 경우, 이 설정을 사용해 커밋되는 주기를 설정
- 더 자주 커밋할수록 오버헤드 역시 늘어나지만 컨슈머가 정지했을 때 발생할 수 있는 중복의 수는 줄어듦

데이터 처리의 신뢰성과 직접 연관되는 것은 아니지만, 만약 컨슈머가 리밸런스를 수행하기 위해 너무 자주 멈추면 신뢰성이 있다고 하기 어려울 것이다. 불필요한 리밸런싱과 리밸런싱이 발생했을 때의 멈춤을 최소화하기 위해서는 어떻게 컨슈머 설정을 해야하는지는 4장에서 다뤘다.

## 7.5.2 컨슈머에서 명시적으로 오프셋 커밋하기

데이터를 신뢰성 있게 다루는 컨슈머를 개발할 때 고려해야 할 중요한 사항들에 대해서 살펴보자.

### 1. 메시지 처리를 먼저, 오프셋 커밋은 나중에

- 폴링 루프에서 모든 처리를 하고 루프 사이의 상태는 저장하지 않는다면 이것은 쉬움
- 자동 오프셋 커밋을 사용하거나, 폴링 루프 끝에서 오프셋 커밋하거나, 루프 안에서 일정한 주기로 오프셋을 커밋해 오버헤드와 중복 처리 회피 사이의 요구 조건의 균형을 맞추던가 하면 됨
- 만약 스레드가 2개 이상 있거나 상태가 있는(stateful) 처리가 필요한 경우, 컨슈머 객체가 스레드 안전하지 않기 때문에 복잡해짐
- 4장에서 이러한 문제를 해결할 수 있는지 예제와 함께 논의함

### 2. 커밋 빈도는 성능과 크래시 발생시 중복 개수 사이의 트레이드오프다

- 커밋 작업은 상당한 성능 오버헤드를 수반
    - `acks=all` 설정과 함께 쓰기 작업을 수행하는 것과 비슷
- 다만 특정 컨슈머 그룹의 모든 오프셋 커밋이 동일한 브로커로 간다는 점과 오버헤드가 된다는 점이 다르다
- 커밋 주기는 성능과 중복 발생의 요구 조건 사이에서 균형을 맞춰야 함
- 메시지를 읽어올 때마다 커밋하는 방식은 매우 낮은 빈도롤 메시지가 들어오는 토픽에나 사용 가능

<aside>

- 커밋 빈도는 성능과 중복 처리 범위 사이의 트레이드오프다.
- 자주 커밋하면 장애 후 중복 처리되는 메시지 수는 줄지만, 커밋 요청이 많아져 처리량이 낮아질 수 있다.
- 드물게 커밋하면 처리량은 좋아지지만, 크래시 발생 시 마지막 커밋 이후 처리한 메시지를 다시 읽게 되어 중복이 늘어날 수 있다.

즉, **오프셋을 자주 커밋할수록 장애 후 다시 처리해야 하는 메시지 수는 줄지만, 커밋 자체의 비용 때문에 성능은 떨어질 수 있다**

</aside>

### 3. 정확한 시점에 정확한 오프셋을 커밋하자

- 폴링 루프 중간에서 커밋할 때 흔히 하는 실수는 마지막으로 처리된 메시지의 오프셋이 아닌, 마지막으로 읽어온 메시지의 오프셋을 커밋하는 것
- 언제나 처리가 완료된 메시지의 오프셋을 커밋하는 것이 중요함
- 즉, 읽기는 했지만 처리되지 않은 메시지의 오프셋을 커밋할 경우 컨슈머 입장에서는 메시지가 누락될 수 있음

### 4. 리밸런스

- 애플리케이션을 설계할 때 컨슈머 리밸런스가 발생할 것이라는 것과 이를 적절히 처리해줘야 한다는 점을 기억해야 함
- 이것은 보통 할당된 파티션이 해제되기 전 오프셋을 커밋하고, 새로운 파티션이 할당되었을 때 애플리케이션이 보유하고 있던 상태를 삭제해주는 작업을 포함

### 5. 컨슈머는 재시도를 해야 할 수도 있다

- 상황에 따라 `poll()` 호출하고 레코드를 처리한 뒤, 일부 레코드는 처리가 완료되지 않아서 나중에 처리되어야 할 수 있음
- 카프카에서 읽어 온 레코드를 데이터베이스에 쓰려고 하는데, 데이터베이스가 사용 가능하지 않아 나중에 재시도해야 할 수 있음
- 전통적인 발행/구독 메시지 전달 시스템과 달리, 카프카 컨슈머들은 오프셋을 커밋할 뿐 각 메시지에 응답을 보내거나 하지 않음
- 레코드 #30 처리에 실패한 상태에서 #31 처리에 성공할 경우, #31 오프셋을 커밋하면 안됨
    - #31 까지의 모든 레코드를 처리했다고 표시하는 것과 같은 결과
- 대신, 다음 두 패턴 중 하나를 시도
    1. 재시도 가능한 에러가 발생했을 경우 마지막으로 처리에 성공한 레코드의 오프셋을 커밋하는 것이다. 그러고 나중에 처리해야 할 레코드들을 버퍼에 저장하고(이렇게 하면 다음 `poll()` 호출에서 덮어쓰지 않음), 컨슈머의 `pause()` 를 호출해 추가적인 `poll()` 호출이 데이터를 리턴하지 않도록 한 뒤 레코드 처리를 계속한다.
    2. 별도의 토픽에 쓴 뒤 계속 진행하는 것이다. 별도의 컨슈머 그룹을 사용해서 재시도 토픽에 저장된 레코드들을 처리하거나, 주 토픽과 재시도 토픽을 모두 구독하는 컨슈머를 하나 둬서 재시도 사이에는 재시도 토픽 구독을 잠시 멈추도록 할 수도 있다. 이 패턴은 데드 레터 큐 시스템과 비슷하다.

### 6. 컨슈머가 상태를 유지해야 할 수도 있다

- 애플리케이션에 따라 `poll()` 메서드 호출 간에 상태를 유지해야 할 수도 있음
- 만약 이동평균(moving average)을 계산하고자 한다면, 카프카에서 새 메시지들을 폴링해 올 때마다 평균값을 업데이트 해줘야 할 것
- 프로세스가 재시작된다면, 단순히 마지막 오프셋부터 읽기 작업을 재개하는 것만으로는 안 되고, 여기에 해당하는 이동평균값을 복구시켜야 할 것
- 이 문제를 해결하는 방법 중 하나는 마지막으로 누적된 값을 애플리케이션이 오프셋을 커밋할 때 `results` 토픽에 쓰는 것
- 이렇게 하면 스레드가 시작될 때, 작업이 중단된 시점과 마지막으로 누적된 값을 가져올 수 있음

# 7.6 시스템 신뢰성 검증하기

설정 검증, 애플리케이션 검증, 프로덕션 환경에 배포된 애플리케이션 모니터링을 통해 세 개의 계층에 걸쳐서 검증을 수행해야 한다.

## 7.6.1 설정 검증하기

- 애플리케이션 로직과 격리된 채 브로커와 클라이언트 설정을 검증하는 것은 아래와 같은 이유로 권장
    - 선택한 구상이 요구 조건을 충족시킬 수 있는지 확인하는 데 도움이 됨
    - 시스템의 예상 작동을 추론해 보기 위한 좋은 방법
        - 카프카는 이러한 검증 작업을 위한 `org.apache.kafka.tools` 패키지에 두 가지 중요한 툴 포함. 명령줄 툴 형태로 자동화된 테스팅 프레임워크에 포함된 형태로든 실행 가능
            - VerifiableProducer
            - VerifiableConsumer
- 검증용 프로듀서를 사용해서 1에서부터 우리가 선택한 값까지의 숫자를 포함하는 메시지를 순서대로 씀
- 검증용 프로듀서에는 일반 프로듀서와 똑같이 `acks`, `retries` , `timeout.ms` 등 설정값을 잡을 수 있고, 메시지 쓰는 속도도 정할 수 있음
- 검증용 프로듀서를 실행시키면, 브로커에 전송된 메시지마다 성공/에러를 출력
- 검증용 컨슈머는 이벤트를 읽어온 순서대로 출력
    - 커밋과 리밸런스에 관련된 정보도 출력
- 어떤 테스트를 할 지 고려해야 함
    - 리더 선출: 리더를 정지시키면 어떻게 될까? 프로듀서와 컨슈머가 평상시처럼 작동을 재개하는데 얼마나 걸릴까?
    - 컨트롤러 선출: 컨트롤러 재시작 후 시스템이 재개되는데 얼마나 걸릴까?
    - 롤링 재시작: 메시지 유실 없이 브로커들을 하나씩 재시작 시킬 수 있을까?
    - 언클린 리더 선출 테스트: 한 파티션의 모든 레플리카들을 하나씩 중단시킨 후 아웃-오브-싱크 상태가 된 브로커를 시작시키면 어떻게 될까? 작업을 재개하려면 어떻게 해야 할까? 용인될 수 있는 수준인가?
- 아파치 카프카 소스코드 저장소는 방대한 테스트 스위트를 포함
    - 스위트 안에 포함된 테스트들은 위에서 설명한 것과 같은 원리로 작동하며, 롤링 업그레이드가 작동하는지 여부를 확인하기 위해 검증용 프로듀서와 컨슈머 사용

## 7.6.2 애플리케이션 검증하기

- 애플리케이션 로직이 카프카의 클라이언트 라이브러리와 상호작용하는 커스텀 에러 처리 코드, 오프셋 커밋, 리밸런스 리스너와 같은 곳들을 확인
- 애플리케이션 로직이 다양해 테스트 방법에 대한 조언은 한계가 있음
- 개발 프로세스의 하나로 애플리케이션의 통합 테스트를 수행할 것을 권장하며, 다양한 장애 상황에 대해 테스트를 수행할 것을 권장
    - 클라이언트가 브로커 중 하나와 연결이 끊김
    - 클라이언트와 브로커 사이의 긴 지연
    - 디스크 꽉 참
    - 디스크 멈춤
    - 리더 선출
    - 브로커 롤링 재시작
    - 컨슈머 롤링 재시작
    - 프로듀서 롤링 재시작
- 아파치 카프카는 장애 주입을 위한 자체적인 프레임워크 트록도르 테스트 프레임워크를 포함
- 상황별로 애플리케이션 예상 작동을 가지고 있을 것인데, 실제로 어떻게 되는지 확인할 수 있도록 테스트를 수행
    - 컨슈머의 롤링 재시작을 테스트 한다면, 컨슈머 리밸런스로 인해 잠시 멈춘 후 읽기가 재개되어 1000개 이하의 중복 메시지를 읽게 될 것이라 예상 가능
    - 이런 테스트를 통해 애플리케이션의 오프셋 커밋과 리밸런스 처리가 제대로 작동하는지 확인 가능

## 7.6.3 프로덕션 환경에서 신뢰성 모니터링하기

- 카프카의 자바 클라이언트들은 클라이언트 쪽 상태와 이벤트를 모니터링할 수 있게 해주는 JMX 지표를 포함
- 프로듀서의 경우 신뢰성 측면에서 가장 중요한 두 지표는 레코드별 에러율과 재시도율
    - 에러율과 재시도율이 올라가는 것은 시스템에 문제가 있음을 가리킬 수 있음
- 이벤트 전송 도중 발생하는 프로듀서 에러 로그도 모니터링 해야함
- 남은 재시도 횟수가 0인 이벤트가 보인다면, 프로듀서에서 재시도 횟수가 고갈된 것
    - 3장에서 프로듀서의 에러 처리를 개선하고 재시도 횟수가 너무 일찍 고갈되는 것을 방지하기 위해 `delivery.timeout.ms` , `retries` 를 어떻게 설정해야 하는지 논의함
- 프로듀서의 ERROR 레벨 로그들은 재시도 불가능한 에러나 재시도 횟수가 고갈된 재시도 가능 에러나, 타임아웃으로 인해 메시지 전송이 완전히 실패했음을 가리킬 가능성이 높음
    - 이 경우 브로커로부터 리턴된 정확한 에러값이 로그에 찍혔을 것
- 컨슈머에서 가장 중요한 지표는 컨슈머 랙(consumer lag)
    - 랙은 컨슈머가 브로커 내 파티션에 커밋된 가장 최신 메시지에서 얼마나 뒤떨어져 있는지 가리킴
    - 이상적인 상황에서는 항상 0이고 계속 최신 메시지만 읽을 것임
    - 현실에서는 `poll()` 호출이 여러 개의 메시지를 리턴하면 컨슈머는 다음 번 메시지를 읽어오기 전까지 리턴된 메시지를 처리하기 때문에, 랙은 오르락내리락 하게 됨
    - 중요한 것은 컨슈머가 뒤쳐지는 게 아니라 계속 따라 붙는다는 것
    - 컨슈머 랙이 오르락내리락 하는 만큼 이 지표에 경보를 설정하는 것은 약간 어려울 수 있음
    - 컨슈머 랙을 확인하기 위해 링크드인에서 개발한 버로우를 사용하면 더 쉽게 할 수 있음
- 카프카 버전 0.10.0 부터 모든 메시지는 이벤트가 생성된 시점의 타임스탬프를 포함
    - 타임스탬프는 이벤트를 전송중인 애플리케이션이나 관련 설정이 잡혀 있는 브로커에 의해 재정의될 수 있음
- 쓰여진 모든 메시지가 적절한 시간 내에 읽히도록 하기 위해 애플리케이션은 자신이 쓴 이벤트의 수를 메시지 형태로 써줘야 할 것이다
    - 컨슈머는 단위 시간당 읽은 이벤트 수와 이벤트가 쓰여진 시점과 읽힌 시점 사이의 랙을 기록해야 할 것
    - 그러고 누락된 메시지가 없음을 확인하기 위해 프로듀서와 컨슈머 양쪽에서 들어온 초당 이벤트 수를 서로 맞춘 뒤 이벤트가 쓰여진 시점과 읽힌 시점 사이의 간격이 적절함을 확인하 수 있도록 해주는 시스템이 필요할 것
    - 이런 종단 모니터링 시스템은 구현하기 쉽지 않을 뿐 아니라 구현 작업에도 시간이 많이 걸릴 수 있음
    - 이런 시스템을 구현한 오픈소스는 없고, 컨플루언트에서 컨트롤 센터의 일부로 제품 형태의 구현제가 나와 있음
- 카프카 브로커는 브로커가 클라이언트로 보내는 에러 응답률을 보여주는 지표들을 포함
    - `kafka.server:type=BrokerTopicMetrics,name=FailedProduceRequestsPerSec`
    - `kafka.server:type=BrokerTopicMetrics,name=FailedFetchRequestsPerSec`
    - 위 두 지표를 수집할 것을 권장
    - 때로는 일정 수준의 에러 응답이 예상되는 경우가 있음
        - 유지 관리 작업을 위해 브로커를 꺼서 다른 브로커가 리더로 선출될 경우, 프로듀서가 `NOT_LEADER_FOR_PARTITION` 에러를 받는 것은 예상된 작동
        - 이 에러를 받은 후에야 메타데이터 업데이트를 요청한 뒤 이벤트 쓰기 작업을 재개할 수 있기 때문
    - 요청 실패가 알 수 없는 이유로 증가할 경우 반드시 원인을 살펴보아야 함
    - 이러한 작업을 편리하게 할 수 있도록 요청 실패 지표에는 브로커가 보낸 에러 응답이 태크 형식으로 달려 있음
