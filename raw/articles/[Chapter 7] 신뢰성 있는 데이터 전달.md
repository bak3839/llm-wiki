# 7.1 신뢰성 보장

- 보장이란, 서로 다른 상황에서도 시스템이 지킬 것이라 보장되는 행동을 의미
    - 데이터베이스의 ACID가 신뢰성 보장의 대표적인 예
- 신뢰성 있는 애플리케이션 개발을 위해 카프카가 제공하는 보장을 이해하는 것이 필수
    - 개발자가 서로 다른 실패 상황에서 시스템이 어떻게 작동하는지 알 수 있게 해주기 때문
- 카프카가 보장하는 것
    - 파티션 안의 메시지들 간에 순서를 보장
        - 나중에 쓰여진 메시지가 먼저 쓰여진 메시지보다 오프셋이 큰 것을 보장
    - 클라이언트가 쓴 메시지는 모든 인-싱크 레플리카의 파티션에 쓰여진 뒤에 커밋된 것으로 간주
        - 굳이 디스트에 flush 될 필요까지는 없음
        - 프로듀서는 메시지가 완전히 커밋된 다음 응답이 올지(acks=all), 리더에게 쓰여진 다음 응답이 올지(acks=1), 네트워크로 전송된 다음 바로 응답이 올지 선택 가능(acks=0)
    - 커밋된 메시지들은 최소 1개의 작동 가능한 레플리카가 남아 있는 한 유실되지 않음
    - 컨슈머는 커밋된 메시지만 읽을 수 있음
- 이런 기본적인 보장들이 신뢰성 있는 시스템을 구축하기 위해 사용될 수 있지만 그 자체로는 완전히 신뢰성 있게 만들어 주지는 못함
- 신뢰성 있는 시스템을 구축하는 데는 트레이드 오프가 있고, 카프카는 트레이드 오프를 조절할 수 있도록 설정 매개변수를 조절함으로써 어느 정도의 신뢰성이 필요한지 결정할 수 있음
    - 트레이드 오프는 메시지를 저장하는 데 있어 신뢰성과 일관성이 우선인지, 가용성과 높은 처리량 + 낮은 지연 + 하드웨어 비용과 같은 다른 고려사항이 우선이냐의 문제인 경우가 많음

# 7.2 복제

- 카프카의 복제 메커니즘은 파티션별로 다수의 레플리카를 유지한다는 특성과 함께 신뢰성 보장의 핵심
- 카프카는 하나의 메시지를 여러 레플리카에 씀으로써 크래시가 나더라도 메시지의 지속성을 유지
- 카프카는 파티션에 저장된 이벤트의 순서를 보장
    - 파티션은 온라인 또는 오프라인 상태일 수 있음
    - 각 파티션은 다수의 레플리카를 가질 수 있고, 그 중 하나가 리더
- 모든 이벤트들은 리더에 쓰여지며 대체로 리더에서 읽혀짐
    - 설정을 통해 팔로워에서 읽도록 변경 가능
- 다른 레플리카들은 리더와 동기화를 맞추면서 최신 이벤트를 제 시간에 복사해오기만 하면 됨
    - 만약 리더가 작동 불능이 되면, 인-싱크 레플리카 중 하나가 새 리더가 됨
- 레플리카는 파티션의 리더거나 아래의 조건을 만족하는 팔로워인 경우 인-싱크 상태로 간주
    - 주키퍼와의 활성 세션 보유. 최근 6초 사이(설정 가능)에 주키퍼로 하트비트 전송함
    - 최근 10초 사이(설정 가능) 리더로부터 메시지를 읽음
    - 최근 10초 사이에 리더로부터 읽어온 메시지들이 가장 최근 메시지. 팔로워가 리더로부터 메시지를 받고 있다는 것만으로 부족. 최근 10초 사이(설정 가능) 랙이 없었던 적이 최소 한 번은 있어야 함
- 다음과 같은 경우엔 아웃-오브 싱크 상태로 간주
    - 레플리카와 주키퍼 사이의 연결 끊김
    - 새 메시지를 읽어오는 것을 중단
    - 최근 10초 동안의 업데이트 내역을 따라오지 못함

<aside>

**아웃-오브-싱크 레플리카**

- 예전 버전의 카프카에서는 1개 이상의 레플리카가 인-싱크 상태와 아웃-오브-싱크 상태를 빠르게 왔다갔다하는 것을 보는 일이 드물지 않았음
    
    - 클러스터에 뭔가가 잘못되어가고 있다는 신호
    - 흔한 원인 중 하나는 최대 요청 크기가 큰 탓에 JVM 힙도 크게 사용중인 경우
    - 이 경우 튜닝을 하지 않으면 브로커의 가비지 수집 시간이 오래 걸려 순간적으로 주키퍼와 연결이 끊어질 수 있음
- 카프카 2.5.0 이후 버전에서 주키퍼 연결 타임아웃과 최대 레플리카 랙 설정값에 기본값을 사용할 경우 이런 문제가 발생하는 것은 굉장히 드물게 됨
    
- 버전 8 이후 JVM에서 G1 GC를 사용할 경우 이 문제를 완화시킬 수 있지만, 여전히 큰 메시지를 사용할 경우 튜닝이 필요 </aside>
    
- 동기화가 풀린 레플리카는 주키퍼와 다시 연결되어 리더에 쓰여진 가장 최근 메시지까지를 따라잡으면 다시 인-싱크 레플리카가 됨
    
- 동기화가 늦은 인-싱크 레플리카는 프로듀서와 컨슈머를 느리게 만들 수 있음
    
    - 프로듀서와 컨슈머는 메시지 커밋되기 전, 모든 인-싱크 레플리카가 해당 메시지를 받을 때까지 기다려야 하기 때문
- 레플리카의 동기화가 풀리면 더 이상 이 레플리카가 메시지를 받을 때까지 기다릴 필요가 없어짐
    
    - 동기화가 밀려 있긴 하지만, 더 이상 성능에 영향을 주지 않음
    - 인-싱크 레플리카 수가 줄어들면 파티션의 실질적인 복제 팩터가 줄어들면서 중단 시간이 길어지거나 데이터 유실 위험성이 높아짐

# 7.3 브로커 설정

토픽 단위에서 신뢰성 트레이드오프를 제어할 수 있다는 것은 신뢰성이 필요한 토픽과 아닌 토픽을 같은 카프카 클러스터에 저장할 수 있음을 의미한다.

## 7.3.1 복제 팩터

- 토픽 단위 설정은 `replication.factor`
- 자동으로 생성되는 토픽에 적용되는 브로커 단위 설정은 `default.replication.factor`
- 카프카의 레플리카 재할당 툴을 사용해 레플리카를 추가/삭제할 수 있음
- 복제 팩터가 N이면 N-1개의 브로커가 중단되어도 토픽에 데이터를 읽거나 쓸 수 있음
    - 복제 팩터가 클수록 가용성과 신뢰성은 향상하고 장애 발생 가능성이 줄어듬
    - 하지만 최소 N개의 브로커가 필요하고, N개의 복사본을 저장하므로 N배의 디스크 공간이 필요
    - 가용성과 하드웨어 사용량 사이에 트레이드오프가 존재
- 그렇다면, 토픽에 몇 개의 레플리카가 적절한지 어떻게 결정할 수 있을까?

### 가용성

레플리카가 하나인 파티션은 정기적으로 브로커를 재시작하면 작동 불능에 빠진다. 레플리카 수가 더 많을수록 가용성은 더 늘어난다.

### 지속성

각 레플리카는 파티션 안의 모든 데이터의 복사본이다. 만약 파티션에 레플리카가 하나뿐이고 어떠한 이유에서 디스크가 사용 불가능하게 될 경우 해당 파티션의 모든 데이터는 유실된다. (특히 서로 다른 저장 장치에) 복사본이 더 많을수록 모든 데이터가 유실될 가능성은 줄어든다.

### 처리량

레플리카가 추가될 때마다 브로커간 트래픽도 늘어난다. 특정 파티션에 10Mbps의 속도로 쓰는데 레플리카가 하나면 복제 트래픽은 하나도 없을 것이다. 레플리카가 2개라면 복제 트래픽은 10Mbps가 되고, 3개 20Mbps, 5개는 40Mbps가 될 것이다. 클러스터의 크기와 용량을 산정할 때는 이를 고려할 필요가 있다.

### 종단 지연

쓰여진 메시지를 컨슈머가 읽을 수 있으려면 모든 인-싱크 레플리카에 복제되어야 한다. 레플리카 수가 더 많을수록 이들 중 하나가 느려져 컨슈머까지 함께 느려질 가능성이 높아진다. 실제로는 어떤 이유레서건 특정 브로커가 느려지면 해당 브로커를 사용하는 모든 클라이언트 역시 복제 팩터와 무관하게 느려질 것이다.

### 비용

더 많은 레플리카를 가질수록 저장소와 네트워크에 들어가는 비용 역시 증가한다. 많은 저장소 시스템들이 각 블록을 3개로 복제해서 저장하는 경우가 많기 때문에 카프카를 설정할 때 복제 팩터를 2로 잡아줌으로써 저장 비용을 줄이는 것이 합리적인 경우가 있다. 단, 복제 팩터가 3인 경우에 비해 여전히 가용성은 줄어든다는 점을 염두에 두자. 지속성이 저장 장치에 의해 보장될뿐이다.

- 레플리카의 위치도 중요
- 카프카는 언제나 같은 파티션의 레플리카들을 서로 다른 브로커에 저장
    - 하지만 이것이 안전하지 않은 경우가 있음
    - 같은 파티션의 모든 레플리카들이 같은 랙(rack)에 설치되어 있는 브로커들에 저장되었는데 랙 스위치가 오작동하면, 복제 팩터하고는 상관없이 해당 파티션은 사용할 수 없게 될 것임
- 랙 단위 사고를 방지하기 위해 브로커들을 서로 다른 랙에 배치한 뒤 `broker.rack` 브로커 설정 매개변수에 랙 이름을 잡아 줄 것을 권장
    - 랙 이름을 설정해주면 카프카는 파티션의 레플리카들이 서로 다른 랙에 분산되어 저장되도록 하여 가용성을 높임
- 클라우드 환경에서 카프카를 운영중일 경우, 가용 영역을 랙과 비슷한 개념으로 사용하는 것이 보통

## 7.3.2 언클린 리더 선출

- 이 설정은 브로커 단위(실제로는 클러스터 단위)에서만 가능
    
    - `unclean.leader.election.ebable` 이고 기본값은 `false` 로 아웃-오브-싱크 레플리카가 리더가 될 수 없도록 설정
- 파티션의 리더가 더 이상 사용 가능하지 않을 경우 인-싱크 레플리카 중 하나가 새 리더가 됨
    
- 커밋된 데이터에 아무런 유실이 없을을 보장한다는 점에서 이런 리더 선출을 ‘클린’하다가 표현
    
    - 커밋된 데이터: 모든 인-싱크 레플리카에 존재하는 데이터
- 하지만 작동 불능에 빠진 리더 외에 인-싱크 레플리카가 없다면 어떻게 될까? 이런 상황은 다음 두 가지 경우 중 하나에서 발생
    
    1. 파티션에 3개의 레플리카가 있고, 팔로워 2개가 작동 불능에 빠진다.(브로커 크래시) 이 경우 프로듀서는 리더에 쓰기 작업을 계속할 것이므로 모든 메시지는 커밋되고 응답이 간다. (리더가 유일한 ISR이기 떄문) 이제 또 브로커가 크래시가 나는 바람에 리더를 사용할 수 없게 되었다고 가정하자. 이 상황에서 아웃-오브-싱크 레플리카 중 하나가 먼저 시작되면, 해당 파티션의 유일한 사용 가능한 레플리카가 아웃-오브-싱크 레플리카가 된다.
    2. 파티션에 3개의 레플리카가 있고, 네트워크 문제가 발생해 팔로워 2개의 복제 작업이 뒤쳐졌다 가정하자. 이 경우 복제 작업은 계속되고 있지만 인-싱크 상태는 아니다. 리더는 유일한 인-싱크 레플리카로 계속 메시지를 받는다. 이제 리더가 작동 불능에 빠지면, 리더가 될 수 있는 레플리카는 아웃-오브-싱크 레플리카밖에 없다.
    
    → 두 가지 경우 모두, 쉽지 않은 결정을 해야 한다.
    
- 만약 아웃-오브-싱크 레플리카가 새 리더가 될 수 없도록 하면, 예전 리더(마지막 인-싱크 레플리카)가 복구될 때까지 해당 파티션은 오프라인 상태
    
- 만약 아웃-오브-싱크 레플리카가 새 리더가 될 수 있도록 한다면, 새 리더가 동기화를 못 한 사이 예전 리더에 쓰여진 모든 메시지들이 유실
    
    - 컨슈머 입장에서의 일관성 역시 어느 정도 깨짐
        - 레플리카 0과1이 사용 불가능한 상황에서 레플리카 2의 오프셋 100~200에 해당하는 메시지를 썼다고 가정 → 2가 리더인 상황
        - 레플리카 2가 작동 불능에 빠지고 레플리카 0이 온라인 상태
        - 레플리카 0은 0~100에 해당하는 메시지만 갖고 있고 100~200에 해당하는 메시지는 없음
        - 0이 리더가 되면 프로듀서는 새 메시지를 이 레플리카에 쓸 것이고 컨슈머 역시 여기서 메시지를 읽어갈 것임
        - 새 리더는 100~200에 예전 리더와 다른 새 메시지들을 가지게 됨
    - 몇몇 컨슈머들은 2가 리더인 상태에서 쓰인 100~200에 해당하는 메시지를 읽었겠지만, 일부는 같은 오프셋을 할당받은 새 메시지를 읽을 것임
        - 일부는 둘을 뒤섰인 채로 읽게 됨
    - 이는 다운스트림 레포트(downstream report)와 같은 경우 꽤나 나쁜 결과를 낼 수 있음
- 정리하면 아웃-오브-싱크 레플리카가 리더가 될 수 있게 허용할 경우 데이터 유실과 일관성 깨짐의 위험성이 있음
    
- 그렇지 않을 경우엔 파티션이 다시 온라인 상태가 될 때까지 원래 리더가 복구되는 것을 기다려야 하는 만큼 가용성이 줄어듬
    

<aside>

만약 언클린 리더 선출을 허용하고, 리더에서 200에 해당하는 메시지 커밋, 리더가 죽고 100까지의 메시지만 가진 언클린 리더가 선출된다면?

</aside>

## 7.3.3 최소 인-싱크 레플리카

- 토픽과 브로커 단위 모두 `min.insync.replicas` 설정 가능
- 카프카가 보장하는 신뢰성에 따르면 데이터는 모든 인-싱크 레플리카에 쓰여진 시점에서 커밋된 것으로 간주
    - ‘모든’이 단 한 개의 레플리카를 의미할 수도 있고, 이 레플리카가 작동 불능에 빠지면 데이터는 유실될 수 있음
- 커밋된 데이터를 2개 이상의 레플리카에 쓰고자 한다면 인-싱크 레플리카의 최솟값을 더 높게 잡아줄 필요가 있음
    - 토픽에 레플리카가 3개 있고 `min.insync.replicas=2` 이면 프로듀서들은 3개의 레플리카 중 최소 2개가 인-싱크 상태인 파티션에만 쓸 수 있음
- 만약 3개 모두 인-싱크 상태면 모든 것이 정상 작동
    - 3개 중 1개만 작동 불능이여도 정상 작동
- 3개 중 2개의 레플리카가 작동 불능이 되면 브로커는 쓰기 요청을 받을 수 없음
    - 데이터 전송을 시도하는 프로듀서는 `NotEnoughReplicasException` 을 받음
    - 컨슈머는 계속 존재하는 데이터를 읽을 수 있음
- 이런 설정에서 인-싱크 레플리카가 하나만 남을 경우 해당 레플리카는 사실상 읽기 전용이 됨
- 이렇게 하면 언클린 리더 선출이 발생하면 사라질 데이터를 쓰거나 읽는 상황을 방지할 수 있음
- 읽기 전용 상태에서 회복하려면 두 개의 사용 불능 레플리카 중 하나를 복구시킨 뒤 리더의 상태를 따라잡아 인-싱크 상태로 들어갈 때까지 기다려야 함

## 7.3.4 레플리카를 인-싱크 상태로 유지하기

- 아웃-오브-싱크 레플리카는 전반적인 신뢰성을 낮추므로 가능한 한 피할 필요가 있음
    
- 레플리카가 아웃-오브-싱크 상태가 되는 두 가지 이유
    
    - 주키퍼와의 연결 끊김
    - 리더 업데이트 내역을 따라가는 데 실패해 복제 랙이 발생
    
    → 카프카는 두 조건에 대한 카프카 클러스터의 민감도를 조절할 수 있는 브로커 설정을 두 개 가지고 있음
    
- `zookeeper.session.timeout.ms` 는 카프카 브로커가 주키퍼로 하트비트 전송을 멈출 수 있는 최대 시간을 정의
    
    - 이 간격 안에만 하트비트를 보내면 주키퍼는 브로커가 죽었다고 판단하지 않아 클러스터에서 제외하지 않음
    - 네트워크 지연의 변동폭이 큰 클라우드 환경에 설치된 카프카 클러스터의 안정성 증대를 위해 6초에서 18초로 증가
    - 일반적으로 이 값은 GC나 네트워크 상황과 같은 무작위적인 변동에 영향을 받지 않을 만큼 높게, 하디만 실제로 작동이 멈춘 브로커가 적시에 탐지될 수 있을 만큼 충분히 낮게 설정될 필요가 있음
- 레플리카가 `replica.lag.time.max.ms` 에 설정된 값보다 더 오랫동안 리더로부터 데이터를 읽어 오지 못하거나, 리더에 쓰여진 최신 메시지를 따라잡지 못하는 경우
    
    - 동기화가 풀린 상태인 아웃-오브-싱크 상태가 됨
    - 기본값은 10초였으나, 클러스터의 회복 탄력성 증대 및 불필요한 변동을 피하기 위해 30초로 조정
        - 이 설정이 컨슈머의 최대 지연에도 영향을 준다는 점을 명심
        - 메시지가 모든 레플리카에 도착해서 컨슈머가 이 메시지를 읽을 수 있게 되는 데 최대 30초가 걸릴 수 있음

## 7.3.5 디스크에 저장하기

- 카프카는 메시지를 받은 레플리카의 수에만 의존할 뿐, 디스크에 저장되지 않은 메시지에 대해서도 응답함
- 카프카는 세그먼트를 교체할 때 (기본값: 1GB)와 재시작 직전에만 메시지를 디스크로 플러시
    - 그 외의 경우에는 리눅스의 페이지 캐시 기능에 의존
        - 페이지 캐시 공간이 다 찼을 경우에만 메시지를 플러시
- 이러한 발상의 배경에는 각각 데이터의 복제본을 가지고 있는, 서로 다른 랙이나 가용 영역에 위치한 세 대의 장비가 리더의 디스크에 메시지를 쓰는 것보다 더 안전하다는 판단이 있음
    - 서로 다른 랙이나 가용 영역에서 동시에 장애가 발생할 가능성이 거의 없기 때문
- 하지만, 브로커가 더 자주 메시지를 저장하도록 설정 가능
    - `flush.messages` : 디스크에 저장되지 않은 최대 메시지 수
    - `flush.ms` : 얼마나 자주 디스크에 메시지를 저장하는지를 조절

<aside>

Kafka의 기본 권장 설정은 **메시지마다 동기적으로 fsync를 호출하지 않고**, OS의 **page cache flush 정책**에 맡긴 뒤 **복제(replication)** 로 내구성을 확보하는 방식이다.

fsync는 파일에 쓴 데이터를 OS page cache에만 두지 않고 실제 디스크까지 강제로 밀어 넣는 시스템 호출이다. 일반적인 단일 노드 저장소에서는 내구성을 위해 중요하지만, Kafka 같은 분산 시스템에서는 내구성의 핵심이 보통 **디스크 강제 flush**가 아니라 **여러 브로커에 복제된 사본**이다.

Confluent 글에서 말하는 Kafka의 관점은 다음과 같다.

- Kafka는 매 쓰기마다 fsync하지 않는 것이 기본이자 권장 설정이다.
- 대신 OS가 page cache의 데이터를 적절히 flush하도록 두고, Kafka는 복제를 통해 장애 내구성을 확보한다.
- 매 메시지마다 fsync를 강제하면 시스템 콜 비용 때문에 성능을 인위적으로 떨어뜨릴 수 있다.
- 특히 작은 producer batch 크기에서는 기본 설정이 매 메시지 fsync보다 훨씬 높은 처리량을 보였다.
- 글의 벤치마크에서는 1KB, 10KB 같은 작은 batch에서 기본 설정이 매 메시지 fsync보다 약 3~5배 높은 처리량을 보였다고 설명한다.
- batch 크기가 커지면 fsync 비용이 여러 메시지에 분산되므로, 기본 설정과 성능 차이가 줄어든다.

**왜 Kafka는 매번 fsync하지 않아도 되는가**

Kafka는 단일 디스크에만 의존하는 저장소가 아니라, 여러 브로커에 데이터를 복제하는 분산 로그다. 따라서 하나의 브로커에서 아직 디스크 flush가 안 된 데이터가 있어도, ISR 복제와 ack 설정을 통해 다른 브로커에 사본이 존재하도록 설계할 수 있다.

</aside>
