# [Chapter 6] 카프카 내부 메커니즘

해당 챕터에 주제들을 깊이 이해하는 것은 카프카를 튜닝할 때 특히 도움이 된다. 튜닝하는 설정이 작동하는 메커니즘을 이해하는 것은 아무렇게 설정값을 바꾸는 것이 아닌, 명확한 의도를 가지고 설정값을 잡아주는 데 큰 도움이 된다.

# 6.1 클러스터 멤버십

- 카프카는 현재 클러스터의 멤버인 브로커들의 목록을 유지하기 위해 아파치 주키퍼를 사용
    - 브로커는 브로커 설정 파일에 정의되거나 자동으로 생성된 고유한 식별자를 보유
    - 브로커 프로세스는 시작될 때마다 주키퍼에 Ephemeral 노드의 형태로 ID 등록
    
    <aside>
    👨‍💻
    
    ephemeral node는 **그 노드를 만든 클라이언트 세션이 살아 있는 동안만 존재하는 znode**
    
    - ZooKeeper의 데이터 노드는 znode라고 부른다.
    - 일반 znode와 달리 ephemeral node는 **생성한 세션에 종속**된다.
    - 세션이 유지되는 동안에는 존재한다.
    - 세션이 종료되면 ZooKeeper가 **자동으로 삭제**한다.
    </aside>
    
- 컨트롤러를 포함한 카프카 브로커들과 몇몇 생태계 툴들은 브로커가 등록되는 주키퍼의 `/brokers/ids` 경로를 구독하여 브로커가 추가/제거될 때마다 알림 받음
    - 동일한 ID를 가진 다른 브로커를 시작한다면 에러 발생
        - 새 브로커는 자신의 ID를 등록하여 하지만, 이미 동일한 브로커 ID를 갖는 ZNode가 있어 실패하기 때문임
- 브로커와 주키퍼 간의 연결이 끊어질 경우(브로커가 정지하거나, 네트워크 단절, 가비지 수집이 오래 걸려 발생할 수 있음)
    - 브로커가 시작될 때 생성한 Ephemeral 노드는 자동으로 주키퍼에서 삭제
    - 이 브로커 목록을 지켜보고 있던(혹은, watch를 설정한) 카프카 컴포넌트들은 해당 브로커가 내려갔음을 알아차리게 됨
- 브로커가 정지하면 브로커를 나타내는 ZNode 역시 삭제
    - 브로커의 ID는 다른 자료구조에 남아 있게 됨
    - 각 토픽의 레플리카 목록에는 해당 레플리카를 저장하는 브로커의 ID가 포함
        - 그렇기 때문에 특정 브로커가 완전히 유실되어 동일한 ID를 가진 새로운 브로커를 투입할 경우, 곧바로 클러스터에서 유실된 브로커의 자리를 대신해 이전 브로커의 토픽과 파티션들을 할당 받음

# 6.2 컨트롤러

- 컨트롤러는 카프카 브로커 기능에 더해 파티션 리더를 선출하는 역할을 추가로 맡음
- 클러스터에서 가장 먼저 시작되는 브로커는 주키퍼의 `/controller` 에 Ephemeral 노드를 생성하여 컨트롤러가 됨
- 다른 브로커도 생성 시도를 하지만 ‘노드가 이미 존재함’ 예외를 받고 클러스터에 이미 컨트롤러가 있다는 것을 알게됨
- 브로커는 주키퍼의 컨트롤러 노드에 뭔가 변동이 생겼을 때 알림을 받기 위해 이 노드에 와치를 설정
    - 와치: 어떤 znode의 상태 변화를 나중에 알려달라고 등록해 두는 일회성 알림 장치
    - 이를 통해 클러스터 안에 한 번에 한 개의 컨트롤러만 있도록 보장
- 컨트롤러 브로커가 멈추거나 주키퍼와의 연결이 끊어진 경우 Ephemeral 노드는 삭제
    - 컨트롤러가 사용하는 주키퍼 클라이언트가 `zookeeper.session.timeout.ms` 에 설정된 값보다 더 오랫동안 하트비트를 전송하지 않는 것도 해당
- Ephemeral 노드가 삭제될 경우 클러스터 안의 다른 브로커들은 주키퍼에 설정된 와치를 통해 컨트롤러가 없어졌다는 것을 알게되고 주키퍼에 컨트롤러 노드를 생성하려고 시도함
    - 가장 먼저 새로운 노드를 생성한 브로커가 다음 컨트롤러가 되고, 존재함 예외 발생 후 와치 설정은 이전과 동일
- 브로커는 새로운 컨트롤러가 선출될 때마다 주키퍼의 조건적 증가 연산에 의해 증가된 에포크(epoch: 세대)를 전달 받음
- 브로커는 현재 컨트롤러의 에포크 값을 알고 있기 때문에 더 낮은 에포크를 가진 컨트롤러로부터 메시지를 받으면 무시
    - 컨트롤러 브로커가 가비지 수집 때문에 멈춘 사이 주키퍼 연결이 끊어질 수 있기 때문에 중요
    - 그 사이에 새 컨트롤러가 선출될 수도 있는 것
    - 이전 컨트롤러가 새로운 컨트롤러가 선출되었다는 것을 알지 못한 채 브로커에 메시지를 보낼 수 있음
    - 이런 컨트롤러를 좀비 컨트롤러라 부름
    - 컨트롤러가 전송하는 메시지에 에포크를 포함하여 브로커는 좀비 컨트롤러가 보낸 메시지를 무시할 수 있음
- 브로커가 컨트롤러가 되면, 클러스터 메타데이터 관리와 리더 선출을 시작하기 전 주키퍼로부터 최신 레플리카 상태 맵을 읽어옴
    - 이 적재 작업은 비동기 API를 사용해 수행되며, 지연을 줄이기 위해 읽기 요청을 여러 단계로 나눠 주키퍼로 전송
    - 파티션 수가 매우 많은 클러스터에서는 적재 작업이 몇 초씩 걸릴 수 있음
- 브로커가 클러스터를 나갔다는 사실을 컨트롤러가 알아차리면, 컨트롤러는 해당 브로커가 리더를 맡고 있던 모든 파티션에 대해 새로운 브로커를 할당해주게 됨
    - 컨트롤러는 새로운 리더가 필요한 모든 파티션을 순회하며 새로운 리더가 될 브로커를 결정
        - 단순히 해당 파티션의 레플리카 목록에서 바로 다음 레플리카가 새 브로커가 됨
        - 그러고 새로운 상태를 주키퍼에 쓴 뒤(요청을 여러 개로 나눠서 비동기 방식으로 전송) 새로 리더가 할당된 파티션의 레플리카를 포함하는 모든 브로커에 `LeaderAndISR` 요청을 전송
        - 이 요청은 해당 파티션들에 대한 새로운 리더와 팔로워 정보를 포함
        - 효율성을 위해 이러한 요청들은 배치 단위로 묶여서 전송
    - 즉, 각각의 요청은 같은 브로커에 레플리카가 있는 다수의 파티션에 대한 새 리더십 정보를 포함하게 되는 것임
- 새로 리더가 된 브로커 각각은 클라이언트로부터의 쓰기 혹은 읽기 요청을 처리하기 시작
    - 반면 팔로워들은 새 리더로부터 메시지를 복제하기 시작
- 클러스터 안의 모든 브로커는 클러스터 내 전체 브로커와 레플리카의 맵을 포함하는 `MetadataCache` 를 가지고 있기 떄문에, 컨트롤러는 모든 브로커에 리더십 변경 정보를 포함하는 `UpdateMetadata` 요청을 보내 각각의 캐시를 업데이트 하도록 함
    - 브로커가 백업을 시작할 때도 비슷한 과정이 반복
    - 차이점은 브로커에 속한 모든 레플리카들은 팔로워로 시작하며, 리더로 선출될 자격을 얻기 위해서는 그 전에 리더에 쓰여진 메시지를  따라잡아야 함

👨‍💻

- 카프카는 컨트롤러를 선출하고 브로커가 클러스터에 들어오거나 나갈 때 컨트롤러에 알려주기 위해 주키퍼의 Ephemeral 노드 기능을 사용
- 컨트롤러는 브로커가 클러스터에 들어오거나 나갈 때 파티션과 레플리카 중에서 리더를 선출할 책임을 짐
- 컨트롤러는 서로 다른 2개의 브로커가 자신이 현재 컨트롤러라 생각하는 스플릿 브레인 현상을 방지하기 위해 에포크 번호를 사용
</aside>

## 6.2.1 KRaft: 카프카의 새로운 래프트 기반 컨트롤러

- KRaft는 새로운 컨트롤러의 프리뷰 버전은 카프카 2.8에 포함되었고, 3.3부터는 정식으로 프로덕션 환경에서 사용 가능한 기능이 됨
- 카프카 커뮤니티는 컨트롤러를 교체하기로 결정했을까?
    1. 컨트롤러가 주키퍼에 메타데이터를 쓰는 작업은 동기적으로 이루어지지만, 브로커 메시지를 보내는 작업은 비동기. 또 주키퍼로부터 업데이트를 받는 과정 역시 비동기적으로 이루어기 때문에 브로커, 컨트롤러, 주키퍼 간에 메타데이터 불일칠가 발생할 수 있고 잡아내기 어려움
    2. 컨트롤러가 재시작될 때마다 주키퍼로부터 모든 브로커와 파티션에 대한 메타데이터를 읽어와야함. 그러고 이 메터데이터를 모든 브로커로 전송. 이 부분이 여전히 병목이 됨. 즉, 파티션과 브로커의 수가 증가함에 따라 컨트롤러 재시작은 더욱 더 느려짐
    3. 메타데이터 소유권 관련 내부 아키텍처는 좋지 못함. 즉, 어떤 작업은 컨트롤러가 하고, 다른 건 브로커가 하고, 나머지는 주키퍼가 직접 수행
    4. 주키퍼는 분산 시스템이며, 카프카와 마찬가지로 운영을 위해서 어느정도 기반 지식이 있어야 함. 그렇기 때문에 카프카를 사용하려는 개발자들은 두 개의 분산 시스템에 대해 배워야 함
    
    → 이런 문제점들을 고려해 카프카 커뮤니티는 현재 사용되는 주키퍼 기반 컨트롤러를 교체하는 쪽을 선택함
    
- 현재 아키텍처에서 주키퍼는 두 개의 중요한 기능을 맡음
    1. 컨트롤러 선출
    2. 클러스터 메타데이터 저장
        - 클러스터 메터데이터: 현재 운영중인 브로커, 설정, 토픽, 파티션, 레플리카 관련 정보
        - 또 컨트롤러 그 자체가 관리하는 메타데이터도 있음
            - 파티션 리더를 선출, 토픽 생성/삭제, 레플리카 할당 시 필요
    
    → 이 모든 기능이 새로운 컨트롤러에서 교체되어야 함
    
- 새로운 컨트롤러 설계의 핵심 아이디어는 카프카 그 자체에 사용자가 상태를 이벤트 스트림으로 나타낼 수 있도록 하는 로그 기반 아키텍처 도입
    - 이 방법의 장점은 카프카 커뮤니티에 익숙한 것으로 다수의 컨슈머를 사용해 이벤트를 재생(replay)함으로써 최신 상태를 빠르게 따라잡을 수 있음
    - 로그는 이벤트 사이에 명확한 순서를 부여하며, 컨슈머들이 항상 하나의 타임라인을 따라 움직이도록 보장
    - 이 장점이 카프카 메타데이터를 관리하는 데에도 적용
- 새로운 아키텍처에서 컨트롤러 노드들은 메터데이터 이벤트 로그를 관리하는 래프트 쿼럼이 됨
    - 이 로그는 클러스터 메타데이터의 변경 내역을 저장
    - 현재 주키퍼에 저장되어 있는 토픽, 파티션, ISR, 설정 등등이 여기에 저장될 것임
- 래프트 알고리즘을 사용하여 컨트롤러 노드들은 외부 시스템에 의존하지 않고 자체적으로 리더를 선출할 수 있게 될 것임
- 메타데이터 로그의 리더 역할을 맡는 컨트롤러는 액티브 컨트롤러라고 함
    - 액티브 컨트롤러는 브로커가 보내온 모든 RPC 호출을 처리
    - 팔로워 컨트롤러들은 액티브 컨트롤러에 쓰여진 데이터들을 복제하며, 액티브 컨트롤러에 장애 발생 시 즉시 투입될 수 있도록 준비 상태를 유지함
    - 컨트롤러들이 모두 최신 상태를 가지고 있으므로, 컨트롤러 장애 복구는 모든 상태를 새 컨트롤러로 이전하는 긴 리로드 시간을 필요하지 않음
- 컨트롤러는 다른 브로커에 변경 사항을 밀어내지(push)하지 않음
    - 대신 다른 브로커들이 새로 도입된 `MetadataFetch API` 를 사용해 액티브 컨트롤러로부터 변경 사항을 당겨옴(pull)
    - 컨슈머의 읽기 요청과 유사하게, 브로커는 마지막으로 가져온 메타데이터 변경 사항의 오프셋을 추적하고 그보다 나중 업데이트만 컨트롤러에 요청
- 브로커는 추후 시동 시간을 줄이기 위해 파티션이 수백만 개에 이를지라도 메타데이터를 디스크에 저장
- 브로커 프로세스는 시작시 주키퍼가 아닌 컨트롤러 쿼럼에 등록
    - 그리고 운영자가 등록을 해제하지 않는 한 이를 유지
    - 브로커가 종료되면 오프라인 상태로 들어가는 것일 뿐 등록은 여전히 유지
    - 온라인 상태지만 최신 메타데이터로 최신 상태를 유지하고 있는 않는 브로커의 경우 펜스된 상태(fenced state)가 되어 클라이언트 요청을 처리할 수 없음
    - `fenced state` : 클라이언트가 더 이상 리더가 아닌, 하지만 최신 상태에서 너무 떨어져 자신이 리더가 아니라는 것을 인식하지 못 하는 브로커에 쓰는 것을 방지

<aside>
👨‍💻

KRaft 도입 이전에 카프카에는 두 종류의 프로세스가 있다.

1. 주키퍼 프로세스: 카프카 클러스터의 동적 메타데이터를 저장한다. 홀수 개의 프로세스가 하나의 쿼럼을 구성하며, 사용자 입장에서 명시적으로 보이지는 않지만 이 중에서 저장된 데이터의 업데이트 작업을 담당하는 리더 프로세스가 하나 있다.
2. 카프카 프로세스: 카프카 데이터를 저장하는 역할을 한다. 이들 중 리더 파티션을 결정하는 역할을 하는 프로세스를 컨트롤러라 한다.

하지만 KRaft 이후부터는 주키퍼 프로세스가 제거되기 때문에 카프카 프로세스 외에 다른 프로세스는 없다. 카프카 프로세스가 다음 두 개 중 하나의 역할을 가지게 된다.

1. 컨트롤러: 카프카 클러스터의 동적 메타데이터를 저장하는 역할을 한다. 1개 이상의 프로세스가 하나의 쿼럼을 구성하며, 이들 중에서 저장된 데이터의 업데이트 및 조회 작업을 담당하는 프로세스를 액티브 컨트롤러라고 한다.
2. 브로커: 카프카 데이터를 저장하는 역할을 한다. 하나의 컨트롤러 쿼럼을 사용하는 브로커들이 모여 하나의 클러스터를 이룬다.

KRaft 이전의 ‘브로커’는 ‘카프카 프로세스’와 동의어이지만, KRaft 이후의 ‘브로커’는 카프카 프로세스가 맞을 수 있는 역할을 의미하는 용어에 더 가깝다.

마찬가지로 KRaft 이전의 ‘컨트롤러’는 브로커 중 파티션 리더를 결정하는 역할을 맡는 어느 특별한 브로커를 가리키지만, KRaft 이후의 ‘컨트롤러’는 동적 메타데이터를 저장하는 역할을 하는 카프카 프로세스를 가리킨다.

</aside>

# 6.3 복제

- 복제는 카프카 아키텍처의 핵심으로 개별적인 노드에 필연적으로 장애가 발생할 수밖에 없는 상황에서 카프카가 신뢰성과 지속성을 보장하는 방식이기 때문
- 카프카에 저장되는 데이터는 토픽을 단위로 해서 조직화
    - 각 토픽은 1개 이상의 파티션으로 분할, 각 파티션은 다시 다수의 레플리카를 가질 수 있음
    - 각각의 레플리카는 브로커에 저장되는데, 대개 하나의 브로커는 서로 다른 토픽과 파티션에 속하는 레플리카를 저장
- 레플리카에는 두 종류가 있음

### 리더 레플리카

각 파티션에는 리더 역할을 하는 레플리카가 하나씩 있다. 일관성을 보장하기 위해 모든 쓰기 요청은 리더로 주어지고, 클라이언트들은 리더나 팔로워로부터 레코드를 읽어올 수 있다.

### 팔로워 레플리카

파티션에 속한 레플리카 중 리더를 제외한 나머지를 팔로워 레플리카라고 한다. 별도로 설정하지 않는 한 팔로워는 클라이언트의 요청을 처리할 수 없다. 팔로워는 주로 리더 레플리카로 들어온 최근 메시지를 복제하여 최신 상태를 유지하는 것이다(ISR). 만약 해당 파티션의 리더에 크래쉬가 날 경우, 팔로워 중 하나가 파티션의 새 리더 파티션으로 승격된다.

<aside>

**팔로워로부터 읽기**

- 해당 기능은 [KIP-392](https://cwiki.apache.org/confluence/display/KAFKA/KIP-392%3A+Allow+consumers+to+fetch+from+closest+replica)에서 추가
- 이 기능의 주 목표는 클라이언트가 리더 대신 가장 가까이 있는 ISR로부터 읽을 수 있게하여 네트워크 트랙픽 비용을 줄이는 것
- 이 기능을 사용하기 위해 클라이언트 위치를 지정하는 `client.rack` 컨슈머 설정값을 잡아주어야 함
- 브로커 설정 중에서 `replica.selector.class` 를 잡아 주어야 함
    - 기본값은 리더로 부터 읽도록 되어 있음
    - `RackAwareReplicaSelector` 로 잡고, 클라이언트의 `client.rack` 과 일치하는 `rack.id` 를 갖는 브로커에 저장된 레플리카로 부터 읽음
    - `ReplicaSelector` 인터페이스로 레플리카 선택 로직을 직접 구현하여 사용 가능
- 복제 프로토콜은 클라이언트가 팔로워 레플리카로부터 메시지를 읽어올 경우에도 커밋된 메시지만 읽도록 확장되었음
    - 즉, 팔로워에서 읽어와도 이전과 동일한 신뢰성이 보장
    - 이를 위해 모든 레플리카에서 리더가 어느 메시지까지 커밋했는지 알아야 함
    - 리더는 팔로워에게 보내는 데이터에 현재의 하이 워터마드(혹은 마지막으로 커밋된 오프셋) 값을 포함해 팔로워에게 알려줌
        - 하이 워터마크 전파에는 약간의 시간이 걸리기 때문에 데이터를 읽은 때는 리더 쪽에서 읽는 편이 팔로워에서 읽는 것보다 빠름

→ 리더에서 읽으면 컨슈머 지연이 줄어드는 장점이 있다는 뜻이므로 이러한 추가 지연을 기억하는 것이 좋음

| **읽기 위치** | **장점** | **단점** |
| --- | --- | --- |
| 리더에서 읽기 | high watermark 전파 지연이 없음 | 리더가 멀리 있으면 네트워크 비용/지연 증가 |
| 가까운 팔로워에서 읽기 | 네트워크 locality가 좋아져 트래픽 비용/지연 감소 가능 | high watermark 전파 지연 때문에 최신 데이터 노출이 조금 늦을 수 있음 |
</aside>

<aside>
👨‍💻

메시지가 ISR에 있는 모든 레플리카에 복제되면 그 메시지는 committed 상태가 되고, 이때 리더의 high watermark가 전진

</aside>

- 리더는 팔로워가 리더의 최신 상태를 유지하고 있는지를 확인하는 일도 수행함
- 팔로워는 새로운 메시지가 도착하는 즉시 리더 레플리카로부터 모든 메시지를 복제해 최신 상태를 유지할 수 있도록 함
    - 다만, 다양한 원인으로 인해 동기화가 깨질 수 있음
        - 네트워크 혼잡으로 인해 복제 속도가 느려짐
        - 브로커가 크래시가 나서 재시작되어 복제 작업을 다시 시작할 수 있을 때까지 해당 브로커에 저장된 모든 레플리카들의 복제 상태가 뒤처지는 사태
- 리더와의 동기화를 유지하기 위해 팔로워들은 리더에 읽기 요청을 보냄
    - 이 요청은 컨슈머가 메시지를 읽어오기 위해 사용하는 그 요청
    - 리더는 메시지를 응답
    - 이 읽기 요청들은 복제를 수행하는 입장에서 다음에 받아야 할 메시지 오프셋 뿐만 아니라 언제나 메시지를 순서대로 반환
    - 즉, 리더 입장에서는 팔로워가 요청한 마지막 메시지까지 복제를 완료했는지, 이후 새로 추가된 메시지가 없는지 알 수 있음
- 리더는 팔로워가 마지막으로 요청한 오프셋 값을 확인해 각 팔로워가 얼마나 뒤쳐져 있는지 알 수 있음
- 만약, 팔로워가 일정 시간 이상 읽기 요청을 보내지 않거나, 요청을 보냈지만 가장 최근에 추가된 메시지를 따라잡지 못하는 경우 해당 레플리카는 동기화가 풀린 것으로 간주
    - 아웃-오브-싱크 레플리카(out-of-sync replica)
- 팔로워가 리더를 따라가는데 실패하면 해당 레플리카는 더 이상 장애 상황에서 리더가 될 수 없음
    - 해당 레플리카가 모든 메시지를 가지고 있지 않기 때문
- 반대로 지속적으로 최신 메시지를 요청하고 있는 레플리카는 인-싱크 레플리카(in-sync replica)라 부름
    - 현재 리더에 장애 발생 시 인-싱크 레플리카만이 파티션 리더로 선출될 수 있음
- 팔로워가 동기화가 풀린 것으로 판정될 때까지 걸리는 시간, 읽기 요청을 보내지 않거나 뒤처진 상태로 있을 수 있는 ‘일정 시간’은 `replica.lag.time.max.ms` 설정 매개변수에 의해 결정
    - 이렇게 허용될 수 있는 랙의 양은 클라이언트의 작동이나 리더 선출 과정에서 데이터 보존에도 영향을 미침
- ‘현재 리더’에 더하여, 각 파티션은 ‘선호 리더(preferred leader)’를 가짐
    - 선호 리더: 토픽이 처음 생성되었을 때 리더였던 레플리카
- 파티션이 처음 생성되던 시점에는 리더가 모든 브로커에 걸쳐 균등하게 분포되기 때문에 ‘선호’라는 표현이 붙음
    - 클러스터 내의 모든 파티션에 대해 선호 리더가 실제 리더가 될 경우 부하가 브로커 사이에 균등하게 분배될 것이라 예상할 수 있음
        - 카프카에는 `auto.leader.rebalance.enable=true` 설정이 기본값
        - 이 설정은 리더가 현재 리더가 아니지만, 현재 리더와 동기화 되고 있을 경우 리더 선출을 실행해 선호 리더를 현재 리더로 만들어 줌

<aside>
👨‍💻

**선호 리더 찾기**

- 현재의 선호 리더를 식별해 내는 가장 좋은 방법은 파티션의 레플리카 목록을 살펴보는 것
    - `kafka-topics.sh` 툴이 출력하는 파티션과 레플리카 상세 정보를 보면 됨
- 목록에 표시된 첫 번째 레플리카가 선호 리더
    - 이것은 현재 리더가 어느 브로커에 있는지, 심지어 레플리카 재할당 툴을 사용해서 다른 브로커로 재할당되었는지 여부와 상관 없음
- 수동으로 레플리카를 재할당하고 있다면 맨 첫 번째로 지정하는 레플리카가 선호 레플리카가 된다는 점을 기억할 필요가 있음

→ 반드시 선호 레플리카를 서로 다른 브로커들로 분산하여 부하가 한쪽으로 몰리지 않도록 하자

</aside>

# **6.4 요청 처리**

- 카프카 브로커의 주요 역할은 클라이언트, 파티션 레플리카, 컨트롤러가 파티션 리더에게 보내는 요청을 처리하는 것이다.
- 카프카는 TCP 위에서 동작하는 이진 프로토콜을 사용한다. 이 프로토콜은 요청의 형식, 요청 처리 성공 시 응답 방식, 오류 발생 시 응답 방식을 정의한다.
- 클라이언트가 연결을 시작하고 요청을 보내면, 브로커는 요청을 처리한 뒤 클라이언트에게 응답한다.
- 특정 클라이언트가 보낸 요청은 브로커가 받은 순서대로 처리된다. 이 특성 때문에 카프카 메시지는 순서가 보장되며, 카프카를 메시지 큐처럼 사용할 수 있다.
- 아파치 카프카 프로젝트는 자바 클라이언트를 포함하며, C, Python, Go 등 여러 언어용 클라이언트도 같은 프로토콜을 사용해 브로커와 통신한다.
- 모든 요청은 공통 헤더를 가진다.
    - 요청 유형: API key라고도 부르며, 어떤 종류의 요청인지 나타낸다.
    - 요청 버전: 서로 다른 버전의 클라이언트 요청을 브로커가 처리할 수 있게 한다.
    - Correlation ID: 요청과 응답을 연결하기 위한 고유 식별자다.
    - Client ID: 요청을 보낸 애플리케이션을 식별한다.
- 브로커는 각 포트마다 연결을 받는 acceptor 스레드를 실행한다. 이 스레드는 클라이언트 연결을 만들고, 요청 처리는 processor 스레드 또는 네트워크 스레드에 넘긴다.
- 네트워크 스레드는 클라이언트 연결에서 요청을 읽어 요청 큐에 넣고, 응답 큐에서 응답을 가져와 클라이언트로 돌려보낸다.
- 요청 큐에 들어간 요청은 I/O 스레드가 가져와 실제 처리를 수행한다.
- 즉시 응답할 수 없는 요청은 완료될 때까지 퍼거토리(purgatory)에 저장된다. 예를 들어 컨슈머 요청은 브로커에 충분한 데이터가 준비될 때까지, 어드민 요청은 토픽 삭제 같은 작업이 끝날 때까지 응답이 지연될 수 있다.

### **쓰기 요청**

- 쓰기 요청은 카프카 브로커로 메시지를 쓰고 있는 프로듀서가 보내는 요청이다.
- 쓰기 요청은 해당 파티션의 리더 레플리카를 가진 브로커로 전달되어야 한다.
- 리더가 아닌 브로커가 쓰기 요청을 받으면 클라이언트는 Not a Leader for Partition 오류를 받는다.
- 클라이언트는 메타데이터 요청을 통해 어떤 브로커가 어떤 파티션의 리더인지 알아낸다.
- 메타데이터 요청은 아무 브로커에나 보낼 수 있으며, 브로커는 토픽, 파티션, 레플리카, 리더 정보를 담은 응답을 반환한다.
- 클라이언트는 메타데이터를 캐시해 두고, 해당 정보를 바탕으로 파티션 리더 브로커에 쓰기 요청을 보낸다.
- 메타데이터는 브로커 추가, 리더 이동, 오류 응답 등에 따라 갱신되어야 한다.

### **읽기 요청**

- 읽기 요청은 카프카 브로커로부터 메시지를 읽어오는 컨슈머나 팔로워 레플리카가 보내는 요청이다.
- 읽기 요청 역시 해당 파티션의 리더 레플리카를 가진 브로커로 전달되어야 한다.
- 클라이언트는 토픽, 파티션, 오프셋 목록을 지정해 특정 위치부터 메시지를 읽어 달라고 요청한다.
- 브로커는 요청받은 오프셋이 유효한지 확인한 뒤 메시지를 반환한다.
- 요청한 오프셋이 너무 오래되어 삭제되었거나 아직 존재하지 않는 오프셋이면 브로커는 오류를 응답한다.
- 카프카는 읽기 응답에서 zero-copy 최적화를 사용한다. 파일 시스템 캐시에서 읽은 데이터를 중간 애플리케이션 버퍼에 복사하지 않고 바로 네트워크 채널로 보낸다.
- 클라이언트는 반환받을 최대 데이터량뿐 아니라 최소 데이터량도 지정할 수 있다. 브로커는 충분한 데이터가 쌓이거나 타임아웃이 발생할 때 응답한다.

## **6.4.1 쓰기 요청**

- acks 설정은 쓰기 작업이 성공한 것으로 간주되기 전에 몇 개의 브로커가 응답해야 하는지를 결정한다.
- acks=1은 리더만 메시지를 받으면 성공으로 본다.
- acks=all은 모든 인-싱크 레플리카가 메시지를 받았을 때 성공으로 본다.
- acks=0은 브로커 응답을 기다리지 않고 메시지가 보내진 시점에 성공으로 간주한다.
- 파티션 리더는 쓰기 요청을 받으면 먼저 요청의 유효성을 검증한다.
    - 데이터를 보내는 사용자가 토픽에 대한 쓰기 권한을 가지고 있는지 확인한다.
    - 요청에 지정된 acks 값이 유효한지 확인한다.
    - acks=all인 경우 메시지를 안전하게 쓸 수 있을 만큼 충분한 인-싱크 레플리카가 있는지 확인한다.
- 유효성 검사를 통과하면 브로커는 새 메시지를 로컬 디스크에 쓴다.
- 리눅스에서는 메시지가 우선 파일 시스템 캐시에 쓰이며, 실제 디스크 반영 시점은 보장되지 않는다. 카프카는 메시지 지속성을 디스크 동기화보다 레플리카 복제에 의존한다.
- 메시지가 파티션 리더에 쓰인 뒤 브로커는 acks 설정에 따라 응답한다.
- acks=0 또는 acks=1이면 브로커는 비교적 빠르게 응답할 수 있다.
- acks=all이면 리더는 요청을 퍼거토리에 저장해 두고, 팔로워 레플리카들이 메시지를 복제한 것을 확인한 뒤 클라이언트에게 응답한다.
- 충분한 수의 레플리카에 복제되지 않은 메시지는 불안전한 것으로 간주된다. 리더 장애가 발생해 다른 레플리카가 리더가 되면, 아직 복제되지 않은 메시지는 사라질 수 있기 때문이다.
- 따라서 acks=all은 쓰기 지연은 늘릴 수 있지만, 메시지가 손실될 가능성을 낮추는 설정이다.

## **6.4.2 읽기 요청**

- 브로커는 쓰기 요청과 유사한 방식으로 읽기 요청을 처리한다.
- 클라이언트는 브로커에 토픽, 파티션, 오프셋 목록을 전달하고, 해당 위치부터 메시지를 보내 달라고 요청한다.
- 클라이언트는 각 파티션에 대해 브로커가 반환할 수 있는 최대 데이터량을 지정한다.
- 최대 데이터량을 지정하는 이유는 브로커가 너무 큰 응답을 보내 클라이언트 메모리를 부족하게 만들지 않기 위해서다.
- 반대로 최소 데이터량도 지정할 수 있다. 예를 들어 최소값을 10KB로 잡으면, 브로커는 보낼 데이터가 최소 10KB 쌓였을 때 결과를 반환한다.
- 최소 데이터량을 설정하면 클라이언트가 너무 자주 요청을 보내고 빈 응답을 받는 상황을 줄일 수 있다. 전체적으로 같은 양의 데이터를 읽더라도 왕복 요청 횟수가 줄어 CPU와 네트워크 사용량이 감소한다.
- 다만 브로커가 충분한 데이터가 모일 때까지 무한정 기다리면 안 되므로, 클라이언트는 타임아웃도 함께 지정한다.
- 타임아웃이 지나면 브로커는 최소 데이터량에 도달하지 못했더라도 현재 보낼 수 있는 데이터를 반환한다.
- 읽기 요청은 요청에 지정된 파티션의 리더 브로커로 전달되어야 한다.
- 파티션 리더는 요청받은 오프셋이 해당 파티션에 존재하는지 확인한다.
- 오프셋이 유효하면 브로커는 파티션에서 지정된 크기 한도만큼 메시지를 읽어 클라이언트에게 보낸다.
- 카프카는 zero-copy를 적용해 파일 시스템 캐시에서 읽은 메시지를 중간 버퍼 없이 네트워크 채널로 보낸다. 이 방식은 복사와 버퍼 관리 오버헤드를 줄여 성능을 높인다.
- 컨슈머가 파티션 리더에 존재하는 모든 데이터를 읽을 수 있는 것은 아니다.
- 일반적으로 컨슈머는 모든 인-싱크 레플리카에 복제된 메시지만 읽을 수 있다.
- 리더는 어떤 메시지가 어느 레플리카까지 복제되었는지 알고 있으며, 모든 인-싱크 레플리카에 복제된 지점까지만 컨슈머에게 노출한다.
- 이 기준 지점을 하이 워터마크라고 볼 수 있다.
- 하이 워터마크 이전 메시지만 읽게 하는 이유는 리더 장애 시 일관성을 지키기 위해서다.
- 리더에만 있고 팔로워에는 아직 복제되지 않은 메시지를 컨슈머가 읽었다면, 리더 장애 후 해당 메시지가 사라져 컨슈머가 읽은 데이터 사이에 불일치가 생길 수 있다.
- 모든 인-싱크 레플리카가 메시지를 받을 때까지 기다리면 읽기 안전성은 높아지지만, 브로커 간 복제가 늦어질 경우 새 메시지가 컨슈머에게 도달하는 시간도 길어진다.
- 이 지연은 replica.lag.time.max.ms 설정과 관련된다. 이 값은 인-싱크 상태로 판단되는 레플리카가 새 메시지를 복제하는 과정에서 지연될 수 있는 최대 시간이다.
- 컨슈머가 많은 파티션에서 이벤트를 읽는 경우, 매번 읽을 파티션 전체 목록과 메타데이터를 요청하고 응답받는 방식은 비효율적이다.
- 카프카는 이런 오버헤드를 줄이기 위해 읽기 세션 캐시(fetch session cache)를 사용한다.
- 컨슈머는 읽고 있는 파티션 목록과 메타데이터를 캐시하는 세션을 만들 수 있다.
- 세션이 생성되면 이후 읽기 요청은 전체 파티션 목록을 매번 보내지 않고 변경분 중심으로 점진적으로 보낼 수 있다.
- 브로커도 변경 사항이 있는 경우에만 응답에 메타데이터를 포함하면 된다.
- 세션 캐시에는 한도가 있기 때문에, 카프카는 팔로워 레플리카나 읽고 있는 파티션 수가 많은 컨슈머를 우선시할 수 있다.
- 어떤 경우에는 캐시된 세션이 생성되지 않거나 제거될 수 있으며, 브로커는 이 상황에 맞는 오류를 반환한다.
- 컨슈머는 오류가 발생하면 전체 파티션 메타데이터를 포함하는 읽기 요청을 다시 보낼 수 있다.

## **6.4.3 기타 요청**

- 지금까지 다룬 요청은 메타데이터, 쓰기, 읽기처럼 가장 일반적인 클라이언트 요청이다.
- 카프카 프로토콜에는 이외에도 컨슈머 그룹 생성, 읽기 오프셋 조정, 컨슈머 그룹 관리, 메타데이터 관리, 보안 관련 요청 등 다양한 요청 유형이 있다.
- 카프카 브로커 사이의 통신에도 동일한 프로토콜이 사용된다.
- 다만 클러스터 내부에서만 사용되는 요청은 일반 클라이언트에서 사용하면 안 된다.
- 예를 들어 컨트롤러가 파티션에 새 리더가 선출되었음을 브로커들에게 알릴 때는 LeaderAndIsr 요청을 사용한다.
- 과거에는 컨슈머가 읽은 오프셋을 기록하기 위해 아파치 주키퍼를 직접 사용했다.
- 이후 카프카 커뮤니티는 OffsetCommit, OffsetFetch, ListOffsets 같은 요청을 추가했고, 클라이언트는 더 이상 주키퍼에 직접 오프셋을 쓰지 않고 카프카에 요청을 보낸다.
- 토픽 생성 같은 메타데이터 관리 작업도 CreateTopic 요청과 AdminClient를 통해 수행할 수 있다.
- 이런 요청들이 카프카 프로토콜의 일부가 되면서, 주키퍼 라이브러리가 없는 언어로 작성된 클라이언트도 브로커에 직접 연결해 토픽 생성 같은 작업을 수행할 수 있게 되었다.
- 카프카는 새 요청 유형을 추가하는 방식뿐 아니라 기존 요청을 수정해 새로운 기능을 추가하기도 한다.
- 요청과 응답에는 버전이 있으므로, 클라이언트와 브로커가 서로 지원하는 요청 버전을 맞추는 것이 중요하다.
- 예를 들어 새 버전 클라이언트가 구버전 브로커에 새 버전 요청을 보내면 브로커가 처리 방법을 알지 못해 오류가 발생할 수 있다.
- 그래서 클라이언트 업그레이드 전에는 브로커를 먼저 업그레이드하는 것이 권장된다. 새 브로커는 예전 요청을 처리할 수 있지만, 예전 브로커는 새 요청을 처리하지 못할 수 있기 때문이다.
- 버전 호환성 문제를 줄이기 위해 ApiVersions 요청이 추가되었다.
- 클라이언트는 ApiVersions 요청으로 연결한 브로커가 어떤 API 버전을 지원하는지 확인하고, 그 브로커와 통신할 때 사용할 수 있는 적절한 요청 버전을 선택할 수 있다.
- 이 방식은 오래된 브로커와 새 클라이언트가 함께 동작해야 하는 환경에서 특히 중요하다.
